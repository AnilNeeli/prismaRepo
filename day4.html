02

Basic operators

OperatorSQL equivalentExample
equals (default) = value published: true
not != value published: { not: true }
in IN (a, b, c) email: { in: ['a@x.com', 'b@x.com'] }
notIn NOT IN (a, b) email: { notIn: ['spam@x.com'] }
lt / lte / gt / gte < <= > >= age: { gte: 18, lte: 60 }
null check IS NULL name: null
not null IS NOT NULL name: { not: null }
typescriptbasic operators
// equals — default, no operator needed
await prisma.post.findMany({ where: { published: true } })

// in — match any value in array
await prisma.user.findMany({
  where: { email: { in: ['a@test.com', 'b@test.com'] } }
})

// range — age between 18 and 60
await prisma.user.findMany({
  where: { age: { gte: 18, lte: 60 } }
})

// null check
await prisma.user.findMany({ where: { name: null } })        // IS NULL
await prisma.user.findMany({ where: { name: { not: null } } }) // IS NOT NULL
03

String operators

typescriptcontains · startsWith · endsWith
// contains — LIKE %value% — anywhere in string
await prisma.user.findMany({
  where: { name: { contains: 'anil' } }
})

// case insensitive — add mode: 'insensitive'
await prisma.user.findMany({
  where: { name: { contains: 'anil', mode: 'insensitive' } }
})

// startsWith — LIKE value%
await prisma.post.findMany({
  where: { title: { startsWith: 'How to' } }
})

// endsWith — LIKE %value
await prisma.post.findMany({
  where: { title: { endsWith: '2025' } }
})
Always use mode insensitive for user-facing search

Without mode: 'insensitive', searching "Prisma" won't find "prisma". In user-facing search, always add it. Note: insensitive mode uses ILIKE in Postgres which can't use standard B-tree indexes — add a GIN index for large datasets.

04

AND / OR / NOT composition

typescriptcomposing conditions
// AND — implicit when you pass multiple fields
await prisma.post.findMany({
  where: { published: true, userId: 'user-123' }  // AND is implicit
})

// explicit AND array
await prisma.post.findMany({
  where: {
    AND: [
      { published: true },
      { title: { contains: 'prisma' } }
    ]
  }
})

// OR — any condition matches
await prisma.user.findMany({
  where: {
    OR: [
      { email: { contains: '@gmail.com' } },
      { email: { contains: '@yahoo.com' } }
    ]
  }
})

// NOT — exclude matches
await prisma.post.findMany({
  where: { NOT: { published: true } }
})

// Combining AND + OR — most common real-world pattern
await prisma.post.findMany({
  where: {
    AND: [
      { published: true },
      {
        OR: [
          { title: { startsWith: 'How' } },
          { content: { contains: 'typescript' } }
        ]
      }
    ]
  }
})
AND is implicit

When you pass multiple fields at the same level, Prisma treats them as AND automatically. You only need the explicit AND: [] array syntax when combining with OR — to control precedence.

05

Relation filters — some, every, none

Filter the parent based on conditions on its children. These are the most powerful and most commonly misunderstood filters.

some

At least one child matches the condition

every

All children match the condition

none

Zero children match the condition

typescriptsome · every · none · is · isNot
// some — at least one child matches
// "users who have at least one published post"
await prisma.user.findMany({
  where: { posts: { some: { published: true } } }
})

// some: {} — empty condition = at least one exists
// "users who have at least one post (any post)"
await prisma.user.findMany({
  where: { posts: { some: {} } }
})

// every — ALL children match
// "users where ALL their posts are published"
await prisma.user.findMany({
  where: { posts: { every: { published: true } } }
})

// none — zero children match
// "users who have NO posts at all"
await prisma.user.findMany({
  where: { posts: { none: {} } }
})

// is / isNot — for nullable single relations (1:1)
// "posts where the author has been deleted"
await prisma.post.findMany({
  where: { user: { is: null } }
})
some/none vs is/isNot — don't mix them up

some / every / none are for array relations (1:N, M:N) — like posts, comments. is / isNot are for nullable single relations (1:1) — like a nullable profile. Using is: null on an array relation throws a type error.

Interview angle

"How do you find all users who have no comments?" — where: { comments: { none: {} } }. The empty {} means "any record" — so none: {} means "no records exist at all".

06

Date filtering

typescriptfiltering by date range
// Last 7 days — use gte not gt
const sevenDaysAgo = new Date()
sevenDaysAgo.setDate(sevenDaysAgo.getDate() - 7)

await prisma.post.findMany({
  where: { createdAt: { gte: sevenDaysAgo } },  // gte includes boundary
  orderBy: { createdAt: 'desc' }
})

// Between two dates
await prisma.post.findMany({
  where: {
    createdAt: {
      gte: new Date('2025-04-01'),
      lte: new Date('2025-04-30')
    }
  }
})
gt vs gte on date ranges

gt (greater than) excludes the boundary — posts created exactly at midnight 7 days ago would be excluded. gte (greater than or equal) includes it. For "last N days" queries, always use gte.

07

Real world fintech query

Combining everything — active users with at least one completed transaction over ₹10,000 in the last 30 days:

typescriptcomplex nested filter
const thirtyDaysAgo = new Date()
thirtyDaysAgo.setDate(thirtyDaysAgo.getDate() - 30)

const users = await prisma.user.findMany({
  where: {
    AND: [
      { isActive: true },
      {
        accounts: {
          some: {
            transactions: {
              some: {
                AND: [
                  { amount: { gte: 10000 } },
                  { status: 'COMPLETED' },
                  { createdAt: { gte: thirtyDaysAgo } }
                ]
              }
            }
          }
        }
      }
    ]
  },
  include: {
    accounts: {
      include: { transactions: true }
    }
  }
})

09

Mistakes made today

  • 1

    prisma.userBlog doesn't exist

    Model is User → client method is prisma.user. Prisma generates camelCase methods from the exact model name. Always match exactly.

  • 2

    Incomplete some syntax — posts: with no value

    Relation filters need an operator. "At least one post exists" = posts: { some: {} }. The empty {} means "any record" — no condition on the child, just checking existence.

  • 3

    Used is: null on array relation

    is / isNot are for nullable single (1:1) relations. For array relations like comments, use none: {} to check "zero exist".

  • 4

    Missing sort direction in orderBy

    orderBy: { createdAt } is invalid shorthand. Always specify direction: orderBy: { createdAt: 'desc' }.

  • 5

    Wrong nesting in OR conditions

    Each OR condition must be its own object with the field at the top level. title and content are separate fields on Post — not nested inside each other.

equals, not, in, notIn, lt/lte/gt/gte, null checks

String ops

contains, startsWith, endsWith. Always add mode: 'insensitive' for user search.

AND is implicit

Multiple fields at same level = AND. Only use explicit AND array when combining with OR.

some / every / none

For array relations. Empty {} = "any record". some: {} = at least one exists.

is / isNot

For nullable single (1:1) relations only. Not for array relations.

Use gte not gt for inclusive boundaries. Always pass a Date object, not a string.
