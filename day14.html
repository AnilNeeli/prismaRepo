01

What is an index?

A separate data structure (B-tree by default) that keeps a sorted copy of one or more columns. Like the index at the back of a book — instead of reading every page, you jump directly to the right page.

sqlwithout index — full table scan
-- Postgres reads EVERY row, checks userId on each one
-- 1 million rows = 1 million comparisons
SELECT * FROM "Post" WHERE "userId" = 'uuid-123'
-- Called: Sequential Scan (Seq Scan)
sqlwith index — direct jump
-- With @@index([userId]) on Post
-- Postgres uses B-tree to jump directly to matching rows
-- 1 million rows = ~20 comparisons (log2 of 1M)
SELECT * FROM "Post" WHERE "userId" = 'uuid-123'
-- Called: Index Scan
02

Adding indexes in Prisma

prisma@@index syntax
model Post {
  id        String   @id @default(uuid())
  userId    String
  published Boolean  @default(false)
  createdAt DateTime @default(now())

  @@index([userId])              // single field index
  @@index([userId, createdAt])   // composite index — order matters
}

// Prisma generates:
// CREATE INDEX "Post_userId_idx" ON "Post"("userId");
// CREATE INDEX "Post_userId_createdAt_idx" ON "Post"("userId", "createdAt");

// @unique and @@unique also create indexes automatically
Indexes that exist automatically

@id → always indexed (primary key). @unique → creates a unique index. @@unique([a, b]) → creates a compound unique index. You only need @@index for non-unique fields you query frequently.

03

Composite index — column order matters

An index can only be used if the query includes the leftmost column(s). Like a phone book sorted by last name then first name — you can find "Smith, John" but can't efficiently find "everyone named John."

prisma@@index([userId, createdAt])
// This composite index helps:
✓ filter by userId only              — leading column present
✓ filter by userId + createdAt       — both columns
✓ filter by userId + order createdAt — leading column present

// This composite index does NOT help:
✗ filter by createdAt only           — leading column missing
uses index where: { userId: 'x' } — leading column present
uses index where: { userId: 'x', createdAt: { gte: date } } — both columns
no index where: { createdAt: { gte: date } } — leading column missing → Seq Scan
Interview question

"Why does @@index([userId, createdAt]) NOT help a query that only filters by createdAt?" — The B-tree is sorted by userId first. Without specifying a userId, Postgres can't use the sorted structure to find records by createdAt alone — it would need to scan all userId groups. A separate @@index([createdAt]) is needed.

04

EXPLAIN ANALYZE — reading query plans

typescripthow to run it
const plan = await prisma.$queryRaw`
  EXPLAIN ANALYZE
  SELECT * FROM "Post" WHERE "userId" = ${'uuid-123'}
`
console.log(JSON.stringify(plan, null, 2))

Seq Scan

Full table scan. Reads every row. Dangerous on large tables. Needs an index.

Bitmap Heap Scan

Partial index usage. Batches multiple row lookups. Acceptable for low-selectivity queries.

Index Scan

Using the index directly. Fast and efficient. What you want to see.

textSeq Scan output — bad
Seq Scan on "Post"
  (cost=0.00..10420.00 rows=1 width=150)
  (actual time=45.231..45.231 rows=1 loops=1)
  Filter: (userId = 'uuid-123')
  Rows Removed by Filter: 99999   ← scanned 100k rows to find 1
textIndex Scan output — good
Index Scan using "Post_userId_idx" on "Post"
  (cost=0.42..8.44 rows=1 width=150)
  (actual time=0.043..0.043 rows=1 loops=1)
  Index Cond: (userId = 'uuid-123')  ← index doing the filtering
What to look for

Seq Scan + large "Rows Removed by Filter" = needs an index. Index Scan + low actual time = working correctly. High cost estimate but low actual time = Postgres estimated wrong but executed well — acceptable.

05

Partial indexes — index only a subset

Index only rows matching a condition. Smaller index, faster lookups, less storage.

bashcreate via custom migration
# Generate SQL without applying
npx prisma migrate dev --name add-partial-index --create-only

# Edit the generated migration.sql:
sqlpartial index — only published posts
-- Only indexes published posts — much smaller than full index
CREATE INDEX "Post_userId_published_idx"
ON "Post"("userId")
WHERE published = true;

-- Only indexes pending transactions
CREATE INDEX "Transaction_status_pending_idx"
ON "Transaction"("accountId")
WHERE status = 'PENDING';
When to use partial indexes

When you frequently query a specific subset of rows — published posts, pending transactions, active users. The index is smaller (faster to build, less memory) and queries on that subset are faster. Prisma doesn't support partial indexes natively — always add via custom migration SQL.

06

Slow query patterns Prisma generates

Pattern 1

Filtering on unindexed relation field

Prisma generates a join + filter on an unindexed column — forces Seq Scan on the related table.

typescript
// ❌ name has no index → Seq Scan on UserBlog
prisma.post.findMany({
  where: { user: { name: { contains: 'anil' } } }
})
// Fix: @@index([name]) on UserBlog
Pattern 2

Ordering on unindexed column

Without an index on the sort column, Postgres sorts all matching rows in memory before returning results.

typescript
// ❌ no index on createdAt → in-memory sort of all rows
prisma.post.findMany({
  orderBy: { createdAt: 'desc' },
  take: 10
})
// Fix: @@index([createdAt]) or @@index([userId, createdAt])
Pattern 3

N+1 on unindexed FK

Even with include, if the FK has no index, the IN clause causes a Seq Scan on the related table.

typescript
// include generates: SELECT * FROM Comment WHERE postId IN (...)
// ❌ if postId has no index → Seq Scan on Comment
prisma.post.findMany({ include: { comments: true } })
// Fix: @@index([postId]) on Comment

Fintech schema — recommended indexes

prismaproduction-ready index setup
model Transaction {
  id             String            @id      // PK — always indexed
  idempotencyKey String            @unique  // unique — auto-indexed
  accountId      String
  status         TransactionStatus
  createdAt      DateTime          @default(now())

  // Most queries filter by account + time range
  @@index([accountId, createdAt])

  // Partial index for pending — via custom migration SQL
  // CREATE INDEX ON "Transaction"("accountId") WHERE status = 'PENDING'
}

model Account {
  id     String @id
  userId String

  @@index([userId])  // findByUserId is the most common query
}

model Post {
  id        String  @id
  userId    String
  published Boolean
  createdAt DateTime

  @@index([userId, createdAt])  // user feed — filter by user, sort by time
}

model Comment {
  id     String @id
  postId String
  userId String

  @@index([postId])  // include comments — postId IN (...) needs this
  @@index([userId])  // find comments by user
}

model AuditLog {
  id        String  @id
  model     String
  recordId  String
  createdAt DateTime

  @@index([model, recordId])  // lookup audit history for a record
  @@index([createdAt])        // time-range queries on audit log
}
08

Slow query workflow

StepAction
1. Spot Enable log: ['query'] — slow queries appear in terminal with high ms
2. Diagnose Run EXPLAIN ANALYZE on the slow query — look for Seq Scan
3. Fix — index Add @@index in schema → migrate dev → rerun EXPLAIN ANALYZE
4. Fix — raw SQL If Prisma's generated SQL is suboptimal — rewrite as $queryRaw
5. Production Use CREATE INDEX CONCURRENTLY via custom migration — no table lock
Interview answer — finding and fixing slow queries

"Enable log: ['query'] to spot slow queries in the terminal. Run EXPLAIN ANALYZE — look for Seq Scan with high rows-removed count. Add @@index for the filtered column, or a composite index if filtering on multiple columns. For production tables use CREATE INDEX CONCURRENTLY via a custom migration to avoid locking the table."

09

Day 15 summary

@@index

Single field or composite. @id, @unique, @@unique all auto-create indexes.

Composite order

Leading column must be in query. @@index([a, b]) helps filter by a or a+b. Not b alone.

EXPLAIN ANALYZE

Seq Scan = bad. Index Scan = good. Bitmap Heap Scan = acceptable. High rows-removed = needs index.

Partial index

WHERE clause on index. Smaller, faster for specific subsets. Add via custom migration SQL only.

Slow patterns

Unindexed relation filter, unindexed orderBy column, unindexed FK in include.

Production

Always CREATE INDEX CONCURRENTLY on large tables — no table lock, zero downtime.
