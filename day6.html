01

Two pagination strategies

Offset (skip/take)

  • Simple — page number × page size
  • Degrades at scale — scans discarded rows
  • Data shifting — records skip or duplicate
  • Good for small datasets, admin panels

Cursor-based

  • Uses last record's id as bookmark
  • Constant speed — uses index, no scanning
  • Stable — inserts don't shift results
  • Best for feeds, infinite scroll, APIs
Interview question — will come up

"Why is cursor pagination better at scale?" — Offset requires Postgres to scan and discard all prior rows. OFFSET 10000 scans 10,000 rows to return 10. Cursor uses an index to jump directly to the right position — same speed on page 1 or page 10,000.

02

Offset pagination — skip/take

typescriptoffset — simple but degrades
// Page 1
const page1 = await prisma.post.findMany({
  take: 10,
  skip: 0,
  orderBy: { createdAt: 'desc' }
})

// Page 2
const page2 = await prisma.post.findMany({
  take: 10,
  skip: 10,
  orderBy: { createdAt: 'desc' }
})

// Dynamic page — formula: skip = (page - 1) * pageSize
async function getPostsPage(page: number, pageSize: number) {
  const [posts, total] = await prisma.$transaction([
    prisma.post.findMany({
      take: pageSize,
      skip: (page - 1) * pageSize,
      orderBy: { createdAt: 'desc' }
    }),
    prisma.post.count()  // total count for page numbers UI
  ])
  return { posts, total, totalPages: Math.ceil(total / pageSize) }
}
Why offset breaks at scale

SQL OFFSET 10000 LIMIT 10 forces Postgres to scan and discard 10,000 rows to return 10. At page 1000 with 10 items per page — that's a 1000x waste. Also, if a new record is inserted between page loads, records at the boundary get skipped or duplicated.

03

Cursor-based pagination

Instead of "skip N rows", cursor says "give me records after this specific record." The cursor is the id of the last item on the current page.

1

First page — no cursor

take: 10, no cursor → returns first 10 records

2

Extract cursor from last item

cursor = posts[posts.length - 1].id

3

Next page — pass cursor + skip: 1

cursor: { id: cursor }, skip: 1 (skips the cursor record itself)

4

Detect last page

if posts.length < take → no more pages, return nextCursor: null

typescriptcursor pagination — reusable function
interface PaginationArgs {
  take: number
  cursor?: string  // id of last item from previous page
}

async function getPostsCursor({ take, cursor }: PaginationArgs) {
  const posts = await prisma.post.findMany({
    take,
    skip: cursor ? 1 : 0,            // skip cursor itself if provided
    cursor: cursor ? { id: cursor } : undefined,
    orderBy: { createdAt: 'desc' },
    select: {
      id: true,
      title: true,
      createdAt: true,
      user: { select: { name: true } }
    }
  })

  return {
    posts,
    nextCursor: posts.length === take
      ? posts[posts.length - 1].id  // more pages exist
      : null                           // last page
  }
}

// Usage
const page1 = await getPostsCursor({ take: 10 })
const page2 = await getPostsCursor({ take: 10, cursor: page1.nextCursor! })
Why skip: 1

The cursor points to the last record you already saw. Without skip: 1, Prisma would include that record again at the top of the next page — duplicating it. skip: 1 tells Prisma to start after the cursor, not at it.

What Postgres actually runs

sqloffset — scans and discards
-- Postgres must scan 10,000 rows to return 10
SELECT * FROM "Post"
ORDER BY "createdAt" DESC
LIMIT 10 OFFSET 10000
sqlcursor — index jump, no scanning
-- Postgres uses index to jump directly to position
SELECT * FROM "Post"
WHERE "id" > 'uuid-last-seen'
ORDER BY "createdAt" DESC
LIMIT 10

05

Sorting — orderBy

typescriptsingle · multi · relation · aggregation
// Single field
prisma.post.findMany({
  orderBy: { createdAt: 'desc' }
})

// Multiple fields — primary sort first, tiebreaker second
prisma.post.findMany({
  orderBy: [
    { published: 'desc' },   // published posts first
    { createdAt: 'desc' }    // then by newest within each group
  ]
})

// Sort by relation field — posts ordered by author name
prisma.post.findMany({
  orderBy: {
    user: { name: 'asc' }
  }
})

// Sort by aggregation — users with most posts first
prisma.userBlog.findMany({
  orderBy: {
    posts: { _count: 'desc' }
  }
})
06

Real world — pagination + filtering + sorting

typescriptproduction feed function
async function getPostsFeed(options: {
  take: number
  cursor?: string
  published?: boolean
  userId?: string
}) {
  const { take, cursor, published, userId } = options

  const posts = await prisma.post.findMany({
    take,
    skip: cursor ? 1 : 0,
    cursor: cursor ? { id: cursor } : undefined,
    where: {
      ...(published !== undefined && { published }),  // optional filter
      ...(userId && { userId })                          // optional filter
    },
    orderBy: { createdAt: 'desc' },
    include: {
      user: { select: { name: true } }
    }
  })

  return {
    posts,
    nextCursor: posts.length === take
      ? posts[posts.length - 1].id
      : null
  }
}
Spread operator for optional filters

...(published !== undefined && { published }) — this pattern conditionally adds a field to the where clause. If published is undefined, nothing is added. If it has a value, { published: true/false } is spread in. Cleaner than writing if/else for every filter.

07

Day 7 summary

skip/take

Simple offset pagination. Degrades at scale. Good for admin panels and small datasets.

Cursor

id of last record as bookmark. Always same speed. Stable under inserts. Use for feeds and APIs.

skip: 1

Always skip the cursor record itself — otherwise it appears at top of next page as a duplicate.

nextCursor

posts.length === take → more pages exist. posts.length < take → last page, return null.

orderBy array

Multiple sort fields — first field is primary sort, rest are tiebreakers in order.

Sort by count

posts: { _count: 'desc' } — sort parent by number of children. No raw SQL needed.
