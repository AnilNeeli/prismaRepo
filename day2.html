01

The golden rule — FK placement

Before anything else, internalize this: the foreign key always lives on the child side — the "many" side, or the "belongs to" side. You will need this instinct for every schema you design.

Rule

Post belongs to User → userId lives on Post, not User. Comment belongs to Post → postId lives on Comment. The parent never holds the child's ID.

Interview question

"Where does the foreign key live in a 1:N relation?" — Always on the child (many) side. The parent holds an array reference in Prisma, but that's virtual — no column in the DB.

02

The three relation types

1:1 — One to One

User has one Profile. Profile belongs to one User.

Key: FK on child side + @unique on the FK field. The @unique is what makes it 1:1 instead of 1:N.

1:N — One to Many

User has many Posts. Post belongs to one User.

Key: FK on child side (Post). Parent (User) has a virtual array posts Post[] — no column in DB.

M:N — Implicit

Post has many Tags. Tag has many Posts. No extra data on the relation.

Key: Prisma auto-creates a hidden join table. You use connect / disconnect. Cannot store extra columns.

M:N — Explicit

Post has many Tags via PostTag. PostTag stores who tagged it and when.

Key: You own the join table as a full model. Use when you need extra data on the relation (assignedAt, role, order, etc.).

03

1:N — One to Many

The most common relation in any schema. A User has many Posts.

prisma1:N example
model User {
  id    String @id @default(uuid())
  email String @unique
  posts Post[]   // virtual — no column in DB. just navigation.
}

model Post {
  id     String @id @default(uuid())
  title  String
  userId String           // actual FK column in DB
  user   User   @relation(fields: [userId], references: [id])
}
What @relation does

fields: [userId] — the FK on THIS model. references: [id] — the PK on the OTHER model. Always read it as: "my userId field references the id field on User."

04

1:1 — One to One

A User has one Profile. The only difference from 1:N is @unique on the FK field.

prisma1:1 example
model User {
  id      String   @id @default(uuid())
  email   String   @unique
  profile Profile?  // optional — user may not have a profile yet
}

model Profile {
  id     String @id @default(uuid())
  bio    String?
  userId String @unique      // @unique = what makes this 1:1 not 1:N
  user   User   @relation(fields: [userId], references: [id])
}
Common mistake

Forgetting @unique on the FK in a 1:1 relation. Without it, Prisma creates a 1:N — multiple profiles per user become possible. The DB won't complain, but your data integrity breaks.

05

M:N — Implicit (Prisma manages the join table)

Use when the relation itself carries no data — just a pure connection between two models.

prismaimplicit M:N
model Post {
  id   String @id @default(uuid())
  tags Tag[]   // just list the other model
}

model Tag {
  id    String @id @default(uuid())
  name  String @unique
  posts Post[]  // just list the other model
}

Prisma auto-creates a hidden _PostToTag table with columns A (postId) and B (tagId). You write to it via connect and disconnect:

typescriptconnecting + disconnecting
// Add a tag to a post
await prisma.post.update({
  where: { id: postId },
  data: { tags: { connect: { id: tagId } } }
})

// Remove a tag from a post
await prisma.post.update({
  where: { id: postId },
  data: { tags: { disconnect: { id: tagId } } }
})
Limitation

The hidden join table only has two columns — the two FKs. You cannot add assignedAt, role, or any extra data. The moment you need metadata on the relation, switch to explicit M:N.

06

M:N — Explicit (you own the join table)

Use when you need extra data on the relation. You create the join table as a full Prisma model.

prismaexplicit M:N — fintech example
model Post {
  id   String    @id @default(uuid())
  tags PostTag[]  // points to join table, not Tag directly
}

model Tag {
  id    String    @id @default(uuid())
  name  String    @unique
  posts PostTag[]  // points to join table, not Post directly
}

model PostTag {               // this IS the join table
  postId     String
  tagId      String
  taggedBy   String           // extra data — who tagged it
  assignedAt DateTime @default(now())

  post Post @relation(fields: [postId], references: [id], onDelete: Cascade)
  tag  Tag  @relation(fields: [tagId], references: [id], onDelete: Cascade)

  @@id([postId, tagId])   // compound PK — one row per post+tag combo
}

With explicit M:N you write directly to the join table:

typescriptwriting to the join table
// Create the relation with extra data
await prisma.postTag.create({
  data: {
    postId: 'post-123',
    tagId: 'tag-456',
    taggedBy: 'anil@test.com',
  }
})

// Query posts with their tags + who tagged them
const post = await prisma.post.findUnique({
  where: { id: 'post-123' },
  include: {
    tags: {
      include: { tag: true }  // nested include to get Tag details
    }
  }
})

Implicit M:N — use when

  • Pure connection, no metadata
  • Simple tags, categories, groups
  • You use connect / disconnect
  • Prisma manages the join table

Explicit M:N — use when

  • Need data on the relation
  • assignedAt, role, order, grade
  • You write to join table directly
  • You need @@id on join table
Interview question — will come up

"When would you use explicit M:N over implicit?" — Explicit when the relation itself carries data. Example: a student enrollment needs a grade and enrolled date — implicit can't store that. A simple post-tag with no metadata → implicit is fine.

07

Referential actions — critical for fintech

What happens to child records when the parent is deleted? Prisma defaults to Restrict — which means deleting a User who has Posts will throw an error. Always be explicit.

prismaonDelete on the relation
model Post {
  userId String
  user   User @relation(fields: [userId], references: [id], onDelete: Cascade)
  // deleting User deletes all their Posts
}
ActionWhat happensUse when
Cascade Delete children when parent deleted Posts when User deleted, Comments when Post deleted
Restrict Block parent delete if children exist (default) Financial records — never delete a User who has transactions
SetNull Set FK to null when parent deleted Optional relations — post's author deleted, post survives with null author
NoAction Like Restrict but checked at end of transaction Complex transactions where order matters
SetDefault Set FK to field's default value Rare — assign to a default "deleted user" record
Fintech interview angle

"Why would you use Restrict on financial records?" — A User with Transactions should never be hard-deleted. Restrict forces you to handle deletion intentionally — soft-delete the user instead. Cascade would silently destroy transaction history.

08

Self-referential relations

A model that relates to itself. Common pattern: Employee → Manager (who is also an Employee).

prismaself-referential
model Employee {
  id         String     @id @default(uuid())
  name       String
  managerId  String?             // nullable — CEO has no manager
  manager    Employee?  @relation("Reports", fields: [managerId], references: [id])
  reports    Employee[] @relation("Reports")
}

The named relation "Reports" is required when a model relates to itself — Prisma needs it to distinguish the two sides.

09

Practice exercise — mistakes + corrections

Built a blogging platform schema. Three mistakes caught:

  • 1

    Duplicate postId field in Comment

    Two postId String declarations in the same model. Prisma throws a validation error. Always check for duplicate field names.

  • 2

    Wrong references on Comment → Post

    Had @relation(fields: [postId], references: [userId]) — referencing userId on Post which doesn't exist. Should be references: [id]. Always reference the PK of the other model.

  • 3

    Missing onDelete: Cascade

    Spec required cascades — Post deleted when User deleted, Comment deleted when Post deleted. Without it Prisma defaults to Restrict, causing runtime errors on delete operations.

Corrected final schema:

prismacorrected blogging schema
model User {
  id        String    @id @default(uuid())
  email     String    @unique
  name      String?
  posts     Post[]
  comments  Comment[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

model Post {
  id        String    @id @default(uuid())
  title     String
  content   String?
  published Boolean   @default(false)
  userId    String
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  user      User      @relation(fields: [userId], references: [id], onDelete: Cascade)
  comments  Comment[]
  tags      PostTag[]
}

model Comment {
  id        String   @id @default(uuid())
  body      String
  postId    String
  userId    String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  user      User     @relation(fields: [userId], references: [id])
  post      Post     @relation(fields: [postId], references: [id], onDelete: Cascade)
}

model Tag {
  id    String    @id @default(uuid())
  name  String    @unique
  posts PostTag[]
}

model PostTag {
  postId     String
  tagId      String
  taggedBy   String
  assignedAt DateTime @default(now())

  post Post @relation(fields: [postId], references: [id], onDelete: Cascade)
  tag  Tag  @relation(fields: [tagId], references: [id], onDelete: Cascade)

  @@id([postId, tagId])
}
10

Day 2 summary

FK rule

Always on the child (many) side. Parent holds a virtual array — no DB column.

1:1 vs 1:N

Only difference: @unique on the FK. Without it you get 1:N.

Implicit M:N

Prisma manages join table. Use connect / disconnect. No extra columns possible.

Explicit M:N

Own the join table. Write directly to it. Use @@id([a, b]) as compound PK.

Referential actions

Default is Restrict. Use Cascade for posts/comments. Use Restrict for financial records.

Self-referential

Model relates to itself. Requires named relation string to disambiguate both sides.
