01

Why soft deletes?

Hard delete — dangerous

  • Record gone forever — unrecoverable
  • Audit trail broken
  • Foreign key violations if referenced
  • Fails regulatory compliance

Soft delete — safe

  • Record stays in DB — recoverable
  • Audit trail intact
  • No FK violations
  • Compliant with financial regulations
Fintech critical

Regulators require transaction history to be permanent. A user deletes their account — their transaction records must still exist for compliance audits. Hard delete in fintech = regulatory violation.

02

Schema setup — deletedAt pattern

prismaadd to every model that needs soft delete
model UserBlog {
  id        String    @id @default(uuid())
  email     String    @unique
  name      String?
  age       Int
  deletedAt DateTime?  // null = active, timestamp = soft deleted
  posts     Post[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

model AuditLog {
  id        String   @id @default(uuid())
  model     String   // "UserBlog", "Post", "Transaction"
  recordId  String   // id of the affected record
  action    String   // "CREATE", "UPDATE", "DELETE"
  before    Json?    // state before the change
  after     Json?    // state after the change
  userId    String?  // who made the change
  createdAt DateTime @default(now())

  @@index([model, recordId])  // fast lookup by record
  @@index([createdAt])        // fast lookup by time
}
Rule

deletedAt = null → active record. deletedAt = timestamp → soft deleted. Always nullable — never a boolean flag. The timestamp tells you when it was deleted, which is valuable for audits.

03

Manual approach — why middleware is better

typescriptmanual — must remember every time
// Soft delete manually
async function softDeleteUser(id: string) {
  return prisma.userBlog.update({
    where: { id },
    data: { deletedAt: new Date() }
  })
}

// Query — must add filter EVERY TIME — miss it once = deleted records appear
async function getActiveUsers() {
  return prisma.userBlog.findMany({
    where: { deletedAt: null }  // ← must remember this everywhere
  })
}
The problem

With the manual approach — every developer on the team must remember where: { deletedAt: null } on every query. Miss it once and soft-deleted records appear in the UI. Middleware solves this by applying the filter automatically everywhere.

04

$use() middleware — how it works

params.model

"UserBlog", "Post", "AuditLog" — which model is being queried

params.action

"create", "update", "delete", "findMany" — which operation

params.args

The actual query args — where, data, include, select

before Intercept query → inspect params.model and params.action → modify if needed
execute await next(params) → actual DB query runs here → returns result
after Use result → write audit log → any post-query logic
return return result → caller gets original query result — must not forget this
05

Soft delete middleware

typescriptglobal soft delete — write once, applies everywhere
prisma.$use(async (params, next) => {

  if (['UserBlog', 'Post'].includes(params.model ?? '')) {

    // Inject deletedAt: null filter on all reads
    if (params.action === 'findMany' || params.action === 'findFirst') {
      params.args.where = {
        ...params.args.where,
        deletedAt: null
      }
    }

    // Convert delete → update (set deletedAt)
    if (params.action === 'delete') {
      params.action = 'update'
      params.args.data = { deletedAt: new Date() }
    }

    // Convert deleteMany → updateMany
    if (params.action === 'deleteMany') {
      params.action = 'updateMany'
      params.args.data = { deletedAt: new Date() }
    }
  }

  return next(params)
})
What this gives you

Now prisma.userBlog.findMany() automatically excludes deleted records. prisma.userBlog.delete() automatically sets deletedAt instead of hard deleting. No developer needs to remember anything.

06

Audit log middleware — explained line by line

typescriptaudit log middleware
prisma.$use(async (params, next) => {
  const writeActions = ['create', 'update', 'updateMany', 'delete', 'deleteMany']

  if (writeActions.includes(params.action)) {

    // Capture BEFORE state — only for update and delete
    // create has no before — record doesn't exist yet
    let before = null
    if (['update', 'delete'].includes(params.action) && params.args.where) {
      const modelName = params.model!.charAt(0).toLowerCase() + params.model!.slice(1)
      before = await (prisma as any)[modelName].findUnique({
        where: params.args.where
      })
    }

    // Execute the actual query
    const result = await next(params)

    // Write audit log AFTER query succeeds
    await prisma.auditLog.create({
      data: {
        model: params.model ?? 'Unknown',
        recordId: result?.id ?? params.args.where?.id ?? 'unknown',
        action: params.action.toUpperCase(),
        before: before ?? undefined,
        after: result ?? undefined,
      }
    })

    return result  // must return — otherwise caller gets undefined
  }

  return next(params)  // reads — pass through unchanged
})
FieldValueWhy
before State captured before query null for creates — record didn't exist. Populated for update/delete.
after result from next() null for deletes — record is gone. Populated for create/update.
recordId result?.id ?? where?.id Three fallbacks — result.id for create/update, where.id for delete
action params.action.toUpperCase() CREATE / UPDATE / DELETE — uppercase for readability
07

Dynamic property access — why (prisma as any)

Inside middleware, the model name is a dynamic string from params.model. TypeScript can't verify a dynamic string is a valid key on PrismaClient — so it throws an error.

GOOD prisma.userBlog — TypeScript knows the type, fully safe
GOOD prisma['userBlog'] — TypeScript can verify the literal string key, safe
ERROR prisma[dynamicString] — TypeScript can't verify at compile time → error
FIX (prisma as any)[dynamicString] — opts out of type check, works at runtime
typescriptconverting model name to client key
// params.model = "UserBlog"
// Prisma client key = "userBlog" (first letter lowercase)

const modelName = params.model!.charAt(0).toLowerCase() + params.model!.slice(1)
// "UserBlog" → "u" + "serBlog" → "userBlog" ✓
// "Post"     → "p" + "ost"     → "post"     ✓

// Then use as any to bypass TypeScript's static check
const record = await (prisma as any)[modelName].findUnique({ where: ... })
One line summary

prisma.userBlog = static key, TypeScript happy. prisma[dynamicString] = dynamic key, TypeScript throws. (prisma as any)[dynamicString] = skip type check, works at runtime. Only use as any when you genuinely need dynamic access.

08

Combined middleware — production pattern

typescriptsoft delete + audit in one middleware
prisma.$use(async (params, next) => {

  // ── SOFT DELETE INTERCEPT ──
  if (['UserBlog', 'Post'].includes(params.model ?? '')) {
    if (params.action === 'findMany' || params.action === 'findFirst') {
      params.args.where = { ...params.args.where, deletedAt: null }
    }
    if (params.action === 'delete') {
      params.action = 'update'
      params.args.data = { deletedAt: new Date() }
    }
    if (params.action === 'deleteMany') {
      params.action = 'updateMany'
      params.args.data = { deletedAt: new Date() }
    }
  }

  // ── AUDIT LOG ──
  const writeActions = ['create', 'update', 'updateMany', 'delete', 'deleteMany']
  let before = null

  if (writeActions.includes(params.action) && params.args?.where?.id) {
    const modelName = params.model!.charAt(0).toLowerCase() + params.model!.slice(1)
    before = await (prisma as any)[modelName].findUnique({
      where: { id: params.args.where.id }
    })
  }

  // Execute query
  const result = await next(params)

  // Write audit log for writes only
  if (writeActions.includes(params.action)) {
    await prisma.auditLog.create({
      data: {
        model: params.model ?? 'Unknown',
        recordId: result?.id ?? 'unknown',
        action: params.action.toUpperCase(),
        before: before ?? undefined,
        after: result ?? undefined,
      }
    })
  }

  return result
})
09

Recovering a soft-deleted record

typescriptfind deleted + restore
// Find soft-deleted records — bypass middleware filter
const deleted = await prisma.userBlog.findFirst({
  where: {
    id: 'user-123',
    deletedAt: { not: null }  // only deleted records
  }
})

// Restore — set deletedAt back to null
await prisma.userBlog.update({
  where: { id: 'user-123' },
  data: { deletedAt: null }
})
Middleware conflict on restore

The soft-delete middleware injects deletedAt: null on findMany/findFirst — so it will never find deleted records. Use findFirst with explicit deletedAt: { not: null } to bypass. The middleware only injects when deletedAt is not already in the where clause.

10

Interview answers

text"how do you implement soft deletes in Prisma?"
"Add a nullable deletedAt DateTime field to the model.
Use $use() middleware to intercept all queries globally —
findMany and findFirst get deletedAt: null injected automatically,
delete and deleteMany get converted to update operations
that set deletedAt to the current timestamp.
Developers never need to remember the filter — applied everywhere."
text"how do you implement audit logging in Prisma?"
"Another $use() middleware that captures the before state
with a findUnique before any write operation, executes the query
via next(params), then writes to an AuditLog table with the
model name, record id, action, before and after states.
Both middlewares run together — soft delete intercept first,
then audit log captures the converted operation."

Day 13 summary

deletedAt

Nullable DateTime. null = active, timestamp = deleted. Never use a boolean flag.

$use() middleware

Intercepts every query. params has model, action, args. next() executes the query.

Global filter

Inject deletedAt: null on findMany/findFirst. Convert delete → update with deletedAt = now().

Audit log

Capture before state before next(). Capture after state from result. Write AuditLog entry.

(prisma as any)

Dynamic model name from params.model. TypeScript can't verify dynamic string keys — use as any.

Restore

findFirst with deletedAt: { not: null }. Update with deletedAt: null to restore.
