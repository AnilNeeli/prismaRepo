01

What is multi-tenancy?

Your app serves multiple customers (tenants) from the same codebase and infrastructure. Each tenant's data must be completely isolated — Tenant A can never see Tenant B's data.

textreal world example
Razorpay serves 500,000 merchants
Each merchant = one tenant
Merchant A's transactions must never appear in Merchant B's dashboard

Row-level

tenantId column on every table. All tenants share one DB.

Most common. Cost efficient. Risk: developer forgets filter.

Schema-per-tenant

Each tenant gets own Postgres schema namespace in one DB.

Moderate isolation. Good for compliance requirements.

Database-per-tenant

Each tenant gets own database entirely.

Maximum isolation. Maximum cost. Enterprise only.

02

Row-level isolation — tenantId middleware

prismaschema — add tenantId to every table
model Transaction {
  id       String @id @default(uuid())
  tenantId String              // every table has this
  amount   Float
  userId   String

  @@index([tenantId])          // always index tenantId
  @@index([tenantId, userId])  // composite for common queries
}
typescriptglobal tenantId middleware
prisma.$use(async (params, next) => {
  const tenantId = getCurrentTenantId()  // from JWT, header, session

  // Inject on reads
  if (['findMany', 'findFirst', 'findUnique'].includes(params.action)) {
    params.args.where = { ...params.args.where, tenantId }
  }

  // Inject on creates
  if (params.action === 'create') {
    params.args.data = { ...params.args.data, tenantId }
  }

  if (params.action === 'createMany') {
    params.args.data = params.args.data.map((item: any) => ({
      ...item, tenantId
    }))
  }

  return next(params)
})
The risk

Middleware is application-level enforcement. If middleware has a bug, wrong tenantId is injected. If someone bypasses Prisma with raw SQL or direct DB access — no protection. If developer forgets to set tenantId context — middleware injects wrong value.

03

Schema-per-tenant — dynamic datasource URL

typescriptdynamic schema per tenant
async function getPrismaForTenant(tenantId: string) {
  const schema = `tenant_${tenantId}`

  return new PrismaClient({
    datasources: {
      db: {
        url: `${process.env.DATABASE_URL}?schema=${schema}`
      }
    }
  })
}

// Usage — queries run in tenant's own schema namespace
const prisma = await getPrismaForTenant('abc')
await prisma.transaction.findMany()
// runs: SELECT * FROM "tenant_abc"."Transaction"
Tradeoffs

Schema-per-tenant gives better isolation than row-level — no risk of cross-tenant data leak from missing filter. But migrations must run per-tenant, which adds complexity. Connection pooling is also harder since each tenant needs their own schema context.

04

Postgres Row-Level Security (RLS)

The most secure row-level isolation — enforced at the DB level, not application level. Used by Supabase.

sqlenable RLS on table
-- Enable RLS on the table
ALTER TABLE "Transaction" ENABLE ROW LEVEL SECURITY;

-- Create policy — only see rows matching current tenantId
CREATE POLICY tenant_isolation ON "Transaction"
  USING (tenant_id = current_setting('app.tenant_id')::uuid);
typescriptset DB context before queries
// Set tenant context on the DB connection
await prisma.$executeRaw`SELECT SET_CONFIG('app.tenant_id', ${tenantId}, true)`

// DB now enforces isolation — even if code has bugs
await prisma.transaction.findMany()  // only returns current tenant's data
// DB applies: WHERE tenant_id = current_setting('app.tenant_id')

Middleware — app enforces

  • Fast to implement
  • Easier to debug
  • Bug in middleware = data leak
  • Bypassing Prisma = no protection
  • Set context in app memory
  • Good for disciplined teams

RLS — DB enforces

  • DB enforces always — no trust needed
  • Works even if app code has bugs
  • Works even with direct DB access
  • More setup required
  • Set context with SET_CONFIG before queries
  • Required for SOC2, PCI-DSS compliance
One line difference

Middleware → app says "only show tenant X data" → DB trusts the app. RLS → DB says "I will only show tenant X data" → no trust needed. For fintech with compliance requirements — RLS is the stronger choice.

Interview answer

"Row-level isolation via middleware is application-level — a bug or Prisma bypass breaks isolation. RLS is database-level — enforced for all connections regardless of how they connect. For a fintech handling payments I'd use RLS for the critical transaction tables and middleware for convenience on non-sensitive tables."

Seeding — prisma/seed.ts

bashinstall faker
npm install @faker-js/faker --save-dev
typescriptprisma/seed.ts — full example
import { PrismaClient } from '@prisma/client'
import { faker } from '@faker-js/faker'

const prisma = new PrismaClient()

async function main() {
  // Clean existing data — order matters due to FK constraints
  await prisma.transaction.deleteMany()
  await prisma.account.deleteMany()
  await prisma.userBlog.deleteMany()

  // Create 10 users
  const users = await Promise.all(
    Array.from({ length: 10 }).map(() =>
      prisma.userBlog.create({
        data: {
          email: faker.internet.email(),
          name: faker.person.fullName(),
          age: faker.number.int({ min: 18, max: 65 }),
        }
      })
    )
  )

  // Create 1 account per user
  const accounts = await Promise.all(
    users.map(user =>
      prisma.account.create({
        data: {
          userId: user.id,
          type: faker.helpers.arrayElement(['SAVINGS', 'CURRENT']),
          balance: faker.number.float({ min: 1000, max: 100000, fractionDigits: 2 }),
        }
      })
    )
  )

  // Create 5 transactions per account
  await Promise.all(
    accounts.flatMap(account =>
      Array.from({ length: 5 }).map(() =>
        prisma.transaction.create({
          data: {
            accountId: account.id,
            amount: faker.number.float({ min: 100, max: 10000, fractionDigits: 2 }),
            type: faker.helpers.arrayElement(['DEBIT', 'CREDIT']),
            status: 'COMPLETED',
            idempotencyKey: faker.string.uuid(),
          }
        })
      )
    )
  )

  console.log('Seeding complete')
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect())

  }
}

Upsert-based seeds — safe to re-run

typescriptcreate — fails on second run
// ❌ throws P2002 on second run — email already exists
await prisma.userBlog.create({
  data: { email: 'admin@test.com', name: 'Admin', age: 30 }
})
typescriptupsert — safe to run multiple times
// ✅ create if not exists, update if exists — always safe
await prisma.userBlog.upsert({
  where: { email: 'admin@test.com' },
  create: { email: 'admin@test.com', name: 'Admin', age: 30 },
  update: { name: 'Admin' }
})
When to use each

Use upsert for fixed seed data — admin users, config records, reference data. Use create with deleteMany cleanup for test/development data where you want a fresh start every run. Delete order must respect FK constraints — children before parents.

Multi-tenancy approaches — comparison

Database-per-tenant Maximum High — per DB cost High Enterprise, data residency

Day 18 summary

Row-level

tenantId on every table. Middleware injects filter. Risk: app-level enforcement only.

Schema-per-tenant

Dynamic DATABASE_URL with ?schema=tenant_x. One DB, separate namespaces.

RLS

DB enforces isolation. SET_CONFIG before queries. No trust in app required. Best for fintech.

Middleware vs RLS

Middleware = app enforces, fast to build. RLS = DB enforces, always safe. Set context differently.

Seeding

faker for realistic data. deleteMany then create for test data. upsert for fixed reference data.

Delete order

Always delete children before parents — FK constraints block parent delete if children exist.
