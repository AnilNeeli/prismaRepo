01

The binary target problem

Prisma generates a query engine — a compiled Rust binary that handles actual DB communication. The binary must match the OS it runs on.

textmismatch = runtime crash
Your dev machine (macOS)   → generates darwin binary
Lambda / ECS (Linux)       → needs linux binary
Mismatch → Prisma throws at runtime — not at build time
prismafix — binaryTargets in schema.prisma
generator client {
  provider      = "prisma-client-js"
  binaryTargets = ["native", "rhel-openssl-3.0.x"]
  //               ↑          ↑
  //               local dev   Lambda / ECS Amazon Linux 2023
}
native Your current machine — macOS, Windows, Linux dev
rhel-openssl-3.0.x Amazon Linux 2023 — Lambda Node 20, ECS Fargate
rhel-openssl-1.0.x Amazon Linux 2 — older Lambda runtimes
linux-arm64-openssl-3.0.x Graviton — ARM-based Lambda/ECS (cheaper, faster)
debian-openssl-3.0.x Ubuntu, Debian based containers
After adding binaryTargets

Run npx prisma generate — regenerates the client with both binaries. The Linux binary gets baked into your Docker image during build. Both binaries add ~60MB to node_modules — acceptable tradeoff.

02

Dockerfile for ECS Fargate

dockerfilemulti-stage build
# Stage 1 — build
FROM node:20-alpine AS builder
WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY prisma ./prisma
RUN npx prisma generate   # generates client + Linux binary

COPY . .
RUN npm run build

# Stage 2 — production image
FROM node:20-alpine AS runner
WORKDIR /app

# Copy only what's needed — keeps image small
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/prisma ./prisma

# Non-root user — security best practice
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

EXPOSE 3000
CMD ["node", "dist/index.js"]
Key points

prisma generate runs during build — binary baked into the image. node_modules copied from builder — includes the generated client. Non-root user — container security best practice. Two-stage build — final image doesn't include build tools.

03

Running migrations in CI/CD

Never run migrate dev in production. The correct deployment sequence:

1

Build Docker image

prisma generate runs here — binary baked in. Push to ECR.

2

Run migrations — separate step

ECS one-off task or entrypoint.sh. npx prisma migrate deploy only.

3

Deploy new ECS task

Only after migration succeeds. Old tasks still running until new ones are healthy.

Separate ECS task — recommended

  • Migration fails → pipeline stops
  • Old app keeps running untouched
  • Users see zero downtime
  • Human watching the pipeline
  • No restart loop risk

entrypoint.sh — simple but riskier

  • Migration + app in one container
  • Crash mid-migration → ECS restart loop
  • Harder to recover
  • OK for staging or simple deploys
  • Never for fintech production
bashPattern A — ECS one-off task (recommended)
# Run migrations as separate ECS task before deploying app
aws ecs run-task \
  --cluster my-cluster \
  --task-definition migrate-task \
  --overrides '{"containerOverrides":[{"name":"app","command":["npx","prisma","migrate","deploy"]}]}'

# Wait for migration to complete before deploying app
aws ecs wait tasks-stopped --cluster my-cluster --tasks $TASK_ARN

# Then deploy the main app
aws ecs update-service --cluster my-cluster --service my-service --force-new-deployment
bashPattern B — entrypoint.sh (simple, riskier)
#!/bin/sh
echo "Running migrations..."
npx prisma migrate deploy

echo "Starting app..."
exec node dist/index.js
04

Why separate migration step — three reasons

  • 1

    Old app keeps serving traffic on failure

    Separate step: migration fails → pipeline stops → old task untouched → users unaffected. Inside startup: ECS starts replacing old tasks, new task crashes, rollback window where some requests hit crashing containers.

  • 2

    No restart loop risk

    entrypoint.sh crashes mid-migration → ECS restarts container → runs migrate deploy again → Prisma detects checksum mismatch → crashes again → ECS loops indefinitely. Separate step runs once in pipeline — no restart risk.

  • 3

    Separation of concerns

    App startup should be fast and predictable. migrate deploy is a one-time infrastructure operation that needs controlled execution, proper logging, alerting, and a human watching the pipeline — not buried in container startup logs.

05

DATABASE_URL via Secrets Manager

Never hardcode connection strings in your image. Inject from AWS Secrets Manager via ECS task definition:

jsonECS task definition — secrets injection
{
  "secrets": [
    {
      "name": "DATABASE_URL",
      "valueFrom": "arn:aws:secretsmanager:us-east-1:123456789:secret:prod/db/url"
    }
  ]
}
// process.env.DATABASE_URL just works in your code — no SDK call needed
Why this approach

No credentials in Docker image or environment variables visible in ECS console. Secrets Manager handles rotation automatically. IAM controls who can read the secret. Works with RDS Proxy IAM auth for zero-credential rotation.

Lambda — complete production setup

typescriptLambda handler — full pattern
import { PrismaClient } from '@prisma/client'

// Outside handler — module cache, survives warm starts
declare global { var prisma: PrismaClient | undefined }

const prisma = global.prisma ?? new PrismaClient({
  datasources: {
    db: { url: process.env.DATABASE_URL + '?connection_limit=1' }
  }
})

if (process.env.NODE_ENV !== 'production') global.prisma = prisma

export const handler = async (event: any) => {
  try {
    const posts = await prisma.post.findMany({ take: 10 })
    return { statusCode: 200, body: JSON.stringify(posts) }
  } catch (error) {
    console.error(error)
    return { statusCode: 500, body: 'Internal server error' }
  }
}
07

Cold start impact

Prisma's query engine binary is ~30MB. Loading it on cold start adds significant latency.

ScenarioCold startSolution
Without Prisma ~200ms —
With Prisma ~800–1200ms Loading 30MB binary
Provisioned concurrency ~0ms Keeps Lambda warm — eliminates cold starts
Prisma Accelerate ~200ms Lightweight HTTP client — no binary
ECS Fargate No cold starts Long-running process — always warm
When cold start matters

User-facing APIs → use provisioned concurrency or move to ECS. Background jobs, event processing, infrequent tasks → cold start acceptable. Report generation, email sending → Lambda is fine.

08

ECS vs Lambda — when to use each

AspectECS FargateLambda
Lifecycle Long-running, always-on Ephemeral, event-driven
Cold starts None 800-1200ms with Prisma
PrismaClient Singleton per task process Singleton per container, global cache
connection_limit usable ÷ task count 1 per Lambda instance
Migrations Separate ECS one-off task Separate Lambda invoke or ECS task
Use for Main API, always-on services Async jobs, reports, emails, webhooks
