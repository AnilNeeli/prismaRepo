01

What is a transaction?

A group of database operations that either all succeed or all fail together. No partial state.

examplewithout transaction — partial state
Debit  ₹1000 from Account A  → success
Credit ₹1000 to Account B    → fails
// Account A lost ₹1000. Account B never received it.
// Money disappeared. Catastrophic.
examplewith transaction — atomic
Debit  ₹1000 from Account A  → success
Credit ₹1000 to Account B    → fails
// Transaction rolls back — Account A debit is reversed
// Both operations cancelled. No money lost.

03

Two transaction styles in Prisma

$transaction([]) — sequential array

  • Operations defined upfront as array
  • Simple syntax
  • Can't read then conditionally write
  • Results of op 1 not available for op 2
  • Use for simple parallel operations

$transaction(callback) — interactive

  • Full async function with tx client
  • Can read → validate → write
  • Throwing any error rolls back all
  • Holds DB connection longer
  • Use for complex business logic
typescriptsequential array — simple ops
const [debit, credit] = await prisma.$transaction([
  prisma.account.update({
    where: { id: 'account-a' },
    data: { balance: { decrement: 1000 } }
  }),
  prisma.account.update({
    where: { id: 'account-b' },
    data: { balance: { increment: 1000 } }
  })
])
typescriptinteractive callback — read then write
const result = await prisma.$transaction(async (tx) => {
  // Step 1 — read current balance
  const accountA = await tx.account.findUnique({
    where: { id: 'account-a' }
  })

  // Step 2 — validate before writing
  if (accountA.balance < 1000) {
    throw new Error('Insufficient funds')  // rolls back everything
  }

  // Step 3 — debit + credit + audit log
  await tx.account.update({
    where: { id: 'account-a' },
    data: { balance: { decrement: 1000 } }
  })
  await tx.account.update({
    where: { id: 'account-b' },
    data: { balance: { increment: 1000 } }
  })
  await tx.auditLog.create({
    data: { action: 'TRANSFER', amount: 1000 }
  })
}, {
  isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
  maxWait: 5000,
  timeout: 10000
})
Interview question — most common

"Difference between $transaction([]) and interactive transaction?" — Array style is sequential but can't read then conditionally write. Interactive callback can read, validate, then write — but holds a DB connection longer. Use interactive for complex business logic like payments.

04

Isolation levels

Controls how much concurrent transactions can see each other's work. Three problems they prevent:

ProblemWhat happens
Dirty read Reading data another transaction wrote but hasn't committed yet — could be rolled back
Non-repeatable read Reading same row twice in one transaction, getting different values — another tx committed a change in between
Phantom read Running same query twice, getting different rows — another tx inserted/deleted rows in between
ReadUncommitted
Prevents nothing — dirty reads possible
Never use in production
fastest
ReadCommitted
Prevents dirty reads only
Postgres default — dashboards, reports
fast
RepeatableRead
Prevents dirty + non-repeatable reads
Inventory checks, read-heavy logic
slower
Serializable
Prevents all anomalies — full isolation
Money transfers, payments — fintech critical
slowest
Why ReadCommitted fails for transfers

Two concurrent transactions both read ₹1000 balance. Both think there's enough. Both debit ₹800. Account goes from ₹1000 to ₹200 instead of being rejected on the second debit. ReadCommitted doesn't prevent this — Serializable does.

How Serializable works

Postgres tracks which rows each transaction read and wrote. Before committing, it checks: "if these ran sequentially, would the result be the same?" If no → aborts one with P2034. You don't write this check — Postgres handles it automatically. Your job is just to catch P2034 and retry.

Interview answer

"Which isolation level for financial transactions?" — Serializable. It ensures transactions run as if sequential, preventing double-spend where two concurrent transfers both read the same balance and both succeed when only one should. Tradeoff: occasional P2034 errors handled with retry + exponential backoff.

05

Idempotency — prevent double charge

Without idempotency — a network retry sends the transfer request twice and the user gets charged twice. The idempotency key is a unique ID per request that ensures the same operation never runs more than once.

typescriptidempotent transfer function
async function transfer(params: {
  fromAccountId: string
  toAccountId: string
  amount: number
  idempotencyKey: string  // unique UUID per request from client
}) {
  return await prisma.$transaction(async (tx) => {

    // Check if already processed
    const existing = await tx.transaction.findUnique({
      where: { idempotencyKey: params.idempotencyKey }
    })
    if (existing) return existing  // already done — return same result

    // Validate balance
    const account = await tx.account.findUnique({
      where: { id: params.fromAccountId }
    })
    if (account.balance < params.amount) {
      throw new Error('Insufficient funds')
    }

    // Debit + Credit
    await tx.account.update({
      where: { id: params.fromAccountId },
      data: { balance: { decrement: params.amount } }
    })
    await tx.account.update({
      where: { id: params.toAccountId },
      data: { balance: { increment: params.amount } }
    })

    // Record with idempotency key
    return await tx.transaction.create({
      data: {
        idempotencyKey: params.idempotencyKey,
        fromAccountId: params.fromAccountId,
        toAccountId: params.toAccountId,
        amount: params.amount,
        status: 'COMPLETED'
      }
    })
  }, { isolationLevel: Prisma.TransactionIsolationLevel.Serializable })
}
06

Retry logic — handling P2034

At Serializable isolation, concurrent transactions can conflict and throw P2034. You need retry logic with exponential backoff.

typescriptretry with exponential backoff
async function withRetry<T>(
  fn: () => Promise<T>,
  retries = 3
): Promise<T> {
  for (let i = 0; i < retries; i++) {
    try {
      return await fn()              // attempt the operation
    } catch (error) {
      if (error.code === 'P2034' && i < retries - 1) {
        // Serialization conflict — wait then retry
        await new Promise(resolve =>
          setTimeout(resolve, 50 * (i + 1))  // 50ms, 100ms, 150ms
        )
        continue
      }
      throw error  // not P2034 or max retries — bubble up
    }
  }
  throw new Error('Max retries exceeded')
}

// Usage — fn is the entire transfer function
const result = await withRetry(() => transfer({
  fromAccountId: 'account-a',
  toAccountId: 'account-b',
  amount: 1000,
  idempotencyKey: 'unique-request-uuid'
}))

Retry flow trace

i = 0 try → run fn() → Postgres detects conflict → throws P2034
wait catch → P2034 → wait 50ms (50 × 1) → continue
i = 1 try → run fn() → Postgres detects conflict → throws P2034
wait catch → P2034 → wait 100ms (50 × 2) → continue
i = 2 try → run fn() → success → return result
done Each retry runs entire function from scratch — fresh read, fresh validate, fresh write
Why idempotency key + retry are both needed

Retry alone → double charge if attempt 1 committed but network dropped before response. Idempotency alone → doesn't handle P2034 conflicts. Together → retry handles DB conflicts, idempotency key prevents double charge on network retries.

07

DB retry vs Kafka retry

DB retry (P2034)

  • Conflict between concurrent DB transactions
  • Nothing committed yet
  • Retry is immediate — milliseconds
  • Same operation, different timing
  • You're trying to make it happen

Kafka retry

  • Async job failed after commit
  • Transaction already in DB
  • Retry can be minutes/hours later
  • Side effects: SMS, webhook, fraud check
  • It happened — you're retrying what follows
typescriptcombined pattern in production
async function transfer(params) {
  // Step 1 — DB transaction with retry (handles P2034)
  const transaction = await withRetry(() =>
    prisma.$transaction(async (tx) => {
      // validate + debit + credit + record
    }, { isolationLevel: Prisma.TransactionIsolationLevel.Serializable })
  )

  // Step 2 — publish event AFTER successful commit
  await kafka.produce('transfer.completed', {
    transactionId: transaction.id,
    fromAccountId: params.fromAccountId,
    toAccountId: params.toAccountId,
    amount: params.amount
  })
  // Kafka consumers retry SMS, webhook, fraud check independently

  return transaction
}
One line summary

DB retry = fix the write before it commits. Kafka retry = fix the side effect after it committed. Two different stages of the same request lifecycle.
