01

Project setup

Use Prisma 5 — not 7. Prisma 7 moved connection config out of schema.prisma into a separate prisma.config.ts and barely any tooling or documentation covers it yet. Every job you'll interview for is on v4 or v5.

bashterminal
mkdir prisma-learn && cd prisma-learn
npm init -y
npm install typescript ts-node @types/node --save-dev
npm install prisma@5 @prisma/client@5 dotenv
npx tsc --init
npx prisma init

prisma init creates two things: prisma/schema.prisma (your schema file) and .env (your connection string). Both are required.

Prisma 7 gotcha — learned the hard way

Prisma 7 removed url from schema.prisma entirely. If you see prisma.config.ts in your project, you're on v7. Stick with v5 for learning — install with prisma@5 @prisma/client@5 explicitly.

02

tsconfig.json — working setup

The default tsconfig.json generated by newer TypeScript versions has settings that conflict with ts-node. Use this config:

jsontsconfig.json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "moduleResolution": "node",
    "esModuleInterop": true,
    "verbatimModuleSyntax": false,
    "strict": true,
    "skipLibCheck": true,
    "ignoreDeprecations": "5.0",
    "types": ["node"]
  }
}
Key changes from default

"module": "CommonJS" — prevents ESM/CJS conflicts with ts-node. "verbatimModuleSyntax": false — turns off the strict import checker. "ignoreDeprecations": "5.0" — silences the node10 moduleResolution deprecation warning.

03

schema.prisma — the correct structure

The schema file has three parts: generator, datasource, and your models. url = env("DATABASE_URL") in the datasource is required — without it Prisma throws on initialization.

prismaprisma/schema.prisma
// generator tells Prisma to create a TypeScript/JS client
generator client {
  provider = "prisma-client-js"   // NOT "prisma-client" — that doesn't exist
}

// datasource tells Prisma which DB and where
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")  // reads from .env — required
}

model User {
  id        String   @id @default(uuid())
  email     String   @unique
  name      String?             // ? = nullable
  age       Int
  balance   Float
  isActive  Boolean  @default(true)
  metadata  Json?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt   // Prisma auto-sets on every update
}

05

Field modifiers — know these cold

ModifierWhat it does
@idPrimary key
@uniqueUnique constraint on a single field
@@unique([a, b])Compound unique — the combination must be unique, not each field individually
@default(uuid())Auto-generate UUID on create. Also: cuid(), now(), autoincrement()
@updatedAtPrisma auto-sets this on every update — you never touch it manually
@map("col")Map Prisma field name to a different DB column name
@@map("table")Map model name to a different DB table name
@ignoreExclude from Prisma Client (field still exists in DB)
Interview question

"What's the difference between @unique and @@unique?" — @unique is on a single field. @@unique([email, tenantId]) means the combination must be unique — one email can exist across multiple tenants.

06

Enums

Enums are stored as strings in Postgres and give you full TypeScript type safety — user.role is typed as "USER" | "ADMIN" | "MODERATOR", not just string.

prisma
enum Role {
  USER
  ADMIN
  MODERATOR
}

model User {
  id   String @id @default(uuid())
  role Role   @default(USER)
}
07

Running your first migration

bash
npx prisma migrate dev --name init

This does three things in order:

  1. Creates a SQL file in prisma/migrations/[timestamp]_init/migration.sql

  2. Applies it to your database

  3. Regenerates the Prisma Client with the latest types

Open the generated migration.sql and read it. Get comfortable with what Prisma produces — it matters when debugging slow queries or fixing failed migrations in production.

Tip

After migrating, run npx prisma studio to open a browser UI at localhost:5555. You can inspect and edit your data visually — useful for verifying migrations worked correctly.

08

First CRUD operations

typescriptsrc/index.ts
import 'dotenv/config'
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient()

async function main() {
  // CREATE
  const user = await prisma.user.create({
    data: { email: 'anil@test.com', name: 'Anil', age: 28, balance: 1000.00 }
  })

  // FIND
  const found = await prisma.user.findUnique({
    where: { email: 'anil@test.com' }
  })

  // UPDATE
  const updated = await prisma.user.update({
    where: { email: 'anil@test.com' },
    data: { balance: 1500.00 }
  })
  console.log('updatedAt changed:', user.updatedAt !== updated.updatedAt) // true

  // DELETE
  await prisma.user.delete({ where: { email: 'anil@test.com' } })
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect())
Gotcha — @updatedAt is client-side

@updatedAt is injected by the Prisma Client, not the database. If you update a row directly via raw SQL or another tool, updatedAt won't change. In production, this means only go through Prisma for writes — don't mix direct DB access.

09

Database setup — Neon.tech

  1. Go to neon.tech → sign up with GitHub → create project named prisma-learn, region US East

  2. Click Connection string → select Prisma → copy the URL

  3. Paste into .env: DATABASE_URL="postgresql://..."

  4. Run npx prisma migrate dev --name init — table appears in Neon dashboard

Important

Verify .env is in your .gitignore before pushing to GitHub. Prisma adds it automatically but always check — a leaked Neon connection string is a live DB credential.

10

Practice exercise — completed

Build a Product model from spec. Solution:

prismacorrect solution
enum Category {
  ELECTRONICS
  CLOTHING
  FOOD
}

model Product {
  id          String   @id @default(uuid())
  name        String
  description String?           // optional — ? required
  price       Float
  sku         String   @unique
  category    Category
  inStock     Boolean  @default(true)
  tags        Json?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
Common mistake

Missing the ? on description — the spec said optional but it's easy to forget. In interviews, always ask "is this field required?" before writing every field.
