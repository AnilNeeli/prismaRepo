class UserRepository {

  async findById(id: string) {
    return prisma.userBlog.findUnique({ where: { id } })
  }

  async findByEmail(email: string) {
    return prisma.userBlog.findUnique({ where: { email } })
  }

  async create(data: Prisma.UserBlogCreateInput) {
    return prisma.userBlog.create({ data })
  }

  async softDelete(id: string) {
    try {
      return await prisma.userBlog.update({
        where: { id },
        data: { deletedAt: new Date() }
      })
    } catch (error) {
      if (error instanceof Prisma.PrismaClientKnownRequestError) {
        if (error.code === 'P2025') throw new Error('User not found')
      }
      throw error
    }
  }

  async restore(id: string) {
    // Must find deleted record first — middleware filters deletedAt: null
    const deleted = await prisma.userBlog.findFirst({
      where: { id, deletedAt: { not: null } }
    })
    if (!deleted) throw new Error('User not found or not deleted')

    return prisma.userBlog.update({
      where: { id },
      data: { deletedAt: null }
    })
  }
}

class AccountRepository {

  async findById(id: string) {
    return prisma.account.findUnique({ where: { id } })
  }

  async findByUserId(userId: string) {
    return prisma.account.findMany({ where: { userId } })
  }

  async create(userId: string, type: AccountType) {
    return prisma.account.create({ data: { userId, type } })
  }

  async updateBalance(id: string, amount: number, type: 'increment' | 'decrement') {
    return prisma.account.update({
      where: { id },
      data: {
        balance: type === 'increment'
          ? { increment: amount }
          : { decrement: amount }
      }
    })
  }

  async softDelete(id: string) {
    try {
      return await prisma.account.update({
        where: { id },
        data: { deletedAt: new Date() }
      })
    } catch (error) {
      if (error instanceof Prisma.PrismaClientKnownRequestError) {
        if (error.code === 'P2025') throw new Error('Account not found')
      }
      throw error
    }
  }
}
updateBalance — why increment/decrement is safe

Prisma's increment and decrement are atomic — Postgres handles the math as balance = balance - 1000. No read-then-write needed. No race condition at the application level.

class TransactionRepository {

  async create(data: Prisma.TransactionUncheckedCreateInput) {
    return prisma.transaction.create({ data })
  }

  async findById(id: string) {
    return prisma.transaction.findUnique({ where: { id } })
  }

  async findByAccountId(accountId: string, cursor?: string, take = 10) {
    return prisma.transaction.findMany({
      where: { accountId },
      take,
      skip: cursor ? 1 : 0,       // skip: 1 skips cursor itself
      cursor: cursor ? { id: cursor } : undefined,
      orderBy: { createdAt: 'desc' }
    })
  }

  async sumByAccountId(accountId: string) {
    // aggregate not groupBy — single account, single total
    const result = await prisma.transaction.aggregate({
      where: { accountId },
      _sum: { amount: true }
    })
    return result._sum.amount ?? 0  // return 0 if no transactions
  }

  async findByIdempotencyKey(key: string) {
    return prisma.transaction.findUnique({
      where: { idempotencyKey: key }
    })
  }
}
aggregate vs groupBy

aggregate — one account, one total. Returns a single number. groupBy — multiple accounts, total per group. Returns an array. For sumByAccountId always use aggregate.

withRetry — the callback wrapper

withRetry takes any async function as a callback and runs it up to 3 times if it fails with P2034. It's a generic shell — works for any async operation.

typescriptwithRetry — generic retry shell
async function withRetry<T>(
  fn: () => Promise<T>,  // function that takes no args, returns Promise
  retries = 3
): Promise<T> {
  for (let i = 0; i < retries; i++) {
    try {
      return await fn()              // execute the function
    } catch (error: any) {
      if (error.code === 'P2034' && i < retries - 1) {
        await new Promise(resolve =>
          setTimeout(resolve, 50 * (i + 1))  // 50ms, 100ms, 150ms
        )
        continue  // retry
      }
      throw error  // P2002, P2025 — don't retry, bubble up
    }
  }
  throw new Error('Max retries exceeded')
}
Why arrow function wrapping matters

withRetry(prisma.$transaction(...)) — executes immediately, can't retry. withRetry(() => prisma.$transaction(...)) — passes a function reference, withRetry controls when it runs. The () => delays execution — that's what makes retrying possible.

i = 0 fn() → $transaction runs → P2034 thrown → rollback
wait catch P2034 → wait 50ms → continue
i = 1 fn() → $transaction runs again from scratch → fresh read → sees updated balance
success return result → caller gets typed Transaction

transferFunds — full implementation

1

Check idempotency key

findUnique by idempotencyKey — if found return existing, don't charge again

2

Validate sender balance

findUnique fromAccount — if not found throw. If balance < amount throw 'Insufficient funds'

3

Debit sender

update fromAccount balance with decrement — atomic, no race condition

4

Credit receiver

update toAccount balance with increment

5

Record DEBIT transaction

create Transaction with type DEBIT, status COMPLETED, idempotencyKey

6

Record CREDIT transaction

create Transaction with type CREDIT, idempotencyKey + '_credit' suffix

typescripttransferFunds — complete
async function transferFunds(params: {
  fromAccountId: string
  toAccountId: string
  amount: number
  idempotencyKey: string
}) {
  return withRetry(() =>
    prisma.$transaction(async (tx) => {

      // Step 1 — idempotency check (inside tx — must be atomic)
      const existing = await tx.transaction.findUnique({
        where: { idempotencyKey: params.idempotencyKey }
      })
      if (existing) return existing

      // Step 2 — validate sender balance
      const fromAccount = await tx.account.findUnique({
        where: { id: params.fromAccountId }
      })
      if (!fromAccount) throw new Error('Source account not found')
      if (fromAccount.balance < params.amount) {
        throw new Error('Insufficient funds')
      }

      // Step 3 — debit sender
      await tx.account.update({
        where: { id: params.fromAccountId },
        data: { balance: { decrement: params.amount } }
      })

      // Step 4 — credit receiver
      await tx.account.update({
        where: { id: params.toAccountId },
        data: { balance: { increment: params.amount } }
      })

      // Step 5 — record debit
      await tx.transaction.create({
        data: {
          accountId: params.fromAccountId,
          amount: params.amount,
          type: 'DEBIT',
          status: 'COMPLETED',
          idempotencyKey: params.idempotencyKey
        }
      })

      // Step 6 — record credit
      return tx.transaction.create({
        data: {
          accountId: params.toAccountId,
          amount: params.amount,
          type: 'CREDIT',
          status: 'COMPLETED',
          idempotencyKey: `${params.idempotencyKey}_credit`
        }
      })

    }, {
      isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
      maxWait: 5000,
      timeout: 10000
    })
  )
}
Idempotency key for credit side

The credit transaction needs a different idempotency key — idempotencyKey + '_credit'. If you used the same key for both, the second create would throw P2002 (unique constraint). Same base key, different suffix per leg of the transfer is the standard pattern.

Why idempotency check is inside the transaction

If the check is outside the transaction, two concurrent requests could both pass the check before either commits — both proceed to charge. Inside the transaction at Serializable isolation, Postgres detects the conflict and aborts one.

07

P2034 — and why retry gets fresh balance

P2034 is thrown when two concurrent Serializable transactions conflict. It's the only Prisma error that's safe to retry automatically.

P2002

Unique constraint violation — duplicate email, duplicate idempotency key

don't retry

P2025

Record not found — update/delete on missing record

don't retry

P2034

Transaction serialization conflict — timing based, next attempt may succeed

safe to retry
Why retry gets updated balance

P2034 means the transaction was rolled back — nothing changed in the DB. The retry runs the entire function from scratch including the balance read. It sees the committed state from the other transaction — the real current balance. This is why retrying from scratch (not resuming) is critical. Fresh read → correct decision.
