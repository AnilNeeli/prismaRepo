01

What is a connection pool?

Opening a new DB connection is expensive — TCP handshake, SSL, Postgres auth = 50–200ms. A pool keeps connections warm and reusable.

Without pool — expensive

  • New connection per query
  • 50-200ms overhead every time
  • Connections discarded after use
  • DB overwhelmed under load

With pool — reused

  • Connections opened once at startup
  • Grabbed from pool for each query
  • Returned to pool after use
  • Near-zero overhead per query
02

PrismaClient singleton — the most important rule

typescriptwrong — new client per request
// ❌ Opens new connection pool on EVERY request
app.get('/posts', async (req, res) => {
  const prisma = new PrismaClient()   // new pool every request
  const posts = await prisma.post.findMany()
  await prisma.$disconnect()
  res.json(posts)
})
// 1000 req/sec = 1000 pools = DB overwhelmed
typescriptcorrect — singleton shared across all requests
// ✅ One pool, created once at startup
const prisma = new PrismaClient()

app.get('/posts', async (req, res) => {
  const posts = await prisma.post.findMany()  // reuses pool
  res.json(posts)
})
How the singleton is saved

Node.js module cache — when a file is first imported, its top-level code runs once and the exports are cached. Every subsequent import in the same process gets the same object. Module level = born when file loads, lives as long as the process. Function level = born on every call, dies when function returns.

03

Lambda singleton — the global cache pattern

Lambda containers are ephemeral. Code outside the handler runs once per container (cold start) and is reused on warm starts.

cold start New container → module loads → new PrismaClient() created → cached
warm start Same container → module already cached → PrismaClient reused → no new connection
cold start New container (concurrent) → fresh module load → new PrismaClient() → new connection
typescriptsrc/lib/prisma.ts — production pattern
import { PrismaClient } from '@prisma/client'

// Tell TypeScript global can have a prisma property
declare global {
  var prisma: PrismaClient | undefined
}

// ?? = use existing if not null/undefined, create new if null
const prisma = global.prisma ?? new PrismaClient({
  datasources: {
    db: { url: process.env.DATABASE_URL + '?connection_limit=1' }
    //                                              ↑
    //                          1 connection per Lambda instance
  }
})

// In dev — cache on global so hot-reload doesn't create new instances
// In prod — global.prisma never set, always creates fresh per container
if (process.env.NODE_ENV !== 'production') {
  global.prisma = prisma
}

export default prisma
Why connection_limit=1 for Lambda

100 concurrent Lambda invocations = 100 containers. With default limit of 10: 100 × 10 = 1000 connections. With limit of 1: 100 × 1 = 100 connections. Predictable and manageable. Use RDS Proxy to multiplex further.

04

connection_limit — how to calculate

ScenarioRDS maxReserveUsableInstancesPer instance
ECS + db.t3.medium 340 40 admin 300 10 tasks 300 ÷ 10 = 30
Lambda + db.t3.micro 85 25 admin 60 60 max concurrent 1 per Lambda
Common mistake

Setting connection_limit=300 on each of 10 ECS tasks = 3000 attempted connections. RDS rejects everything. Always divide total usable connections by number of instances/tasks running simultaneously.

05

PgBouncer — the connection pooler

Sits between your app and Postgres. Takes many app connections, forwards only a few to Postgres. Like a hotel receptionist — handles 1000 guests but only sends 20 to the manager at a time.

textarchitecture
Your App (100 connections) → PgBouncer → Postgres (20 connections)
                                  ↑
                     queues requests when all 20 are busy
                     app never sees connection refused
Session mode safe

One Postgres connection per client session. Released when client disconnects. Transactions work fine — same connection throughout. Less efficient — 100 clients = 100 Postgres connections.

Transaction mode efficient but needs pgbouncer=true

One Postgres connection per transaction. Released immediately after COMMIT. 1000 app connections → 20 Postgres connections. Breaks Prisma interactive transactions unless pgbouncer=true is set.

Statement mode never use with Prisma

One Postgres connection per query. Most aggressive. Breaks transactions, prepared statements, session variables. Never use with Prisma.

06

Why transaction mode breaks Prisma — with real example

10 users making transfers. PgBouncer has 3 Postgres connections (C1, C2, C3).

Transaction mode — connection assignment timeline

t=0 User 1 sends BEGIN → PgBouncer assigns C1
t=1 User 2 sends BEGIN → PgBouncer assigns C2
t=2 User 3 sends BEGIN → PgBouncer assigns C3
t=3 User 3 sends COMMIT → C3 returned to pool
t=4 User 4 sends BEGIN → PgBouncer assigns C3 (just freed)
t=5 User 1 sends COMMIT → C1 returned to pool

This works fine for simple queries. The problem is with Prisma's interactive transactions:

Prisma $transaction — breaks in transaction mode

t=0 Prisma sends BEGIN → PgBouncer assigns C1
t=1 Prisma sends SELECT balance → C1 used, result returned
t=2 PgBouncer thinks query done → releases C1 back to pool
t=3 Prisma sends UPDATE balance → PgBouncer assigns C2 (different connection!)
t=4 Postgres C2 has no open transaction → UPDATE auto-commits without validation check
t=5 Prisma sends COMMIT → PgBouncer assigns C3 → "no open transaction" error
Real world consequence

Balance check ran on C1. UPDATE ran on C2 (auto-committed, no transaction). The validation and the write were on different connections — never in the same transaction. Money can go negative. Data corrupted silently.

textfix — pgbouncer=true in connection string
DATABASE_URL="postgresql://user:pass@pgbouncer:6432/db?pgbouncer=true"

// pgbouncer=true tells Prisma to use SET LOCAL instead of BEGIN/COMMIT
// PgBouncer recognizes SET LOCAL and pins the connection
// for the entire transaction — same connection throughout

08

idle in transaction — detecting and fixing

A transaction was opened (BEGIN sent) but never committed or rolled back. The connection is poisoned — if reused from the pool, queries run inside an old uncommitted transaction.

typescriptthe bug — missing ROLLBACK in catch
async function transfer() {
  await prisma.$executeRaw`BEGIN`

  try {
    await prisma.$executeRaw`UPDATE "Account" SET balance = balance - 1000 ...`
    throw new Error('Something failed')   // error thrown
    await prisma.$executeRaw`COMMIT`     // never reached
  } catch (error) {
    // ❌ forgot ROLLBACK — connection returned to pool with BEGIN still open
    throw error
  }
}
// Connection state: "idle in transaction" — poisoned
1

Detect

SELECT pid, state, now() - state_change as duration FROM pg_stat_activity WHERE state = 'idle in transaction'

2

Kill stuck connection

SELECT pg_terminate_backend(pid) — terminates the specific stuck connection

3

Fix root cause

Use prisma.$transaction() — handles COMMIT and ROLLBACK automatically, can't forget

4

Prevent automatically

Set timeout: 10000 on $transaction — Prisma kills transactions that take too long

Why $transaction prevents this

Prisma's $transaction() wraps everything in try/catch internally. If anything throws — Prisma automatically sends ROLLBACK before rethrowing. You physically can't forget the ROLLBACK because you never write it manually.

Day 16 summary

Singleton

One PrismaClient per process. Module cache saves it. Never instantiate inside a request handler.

connection_limit

Total usable ÷ number of instances. Leave headroom for admin. 1 per Lambda instance.

PgBouncer modes

Session = safe. Transaction = efficient but needs pgbouncer=true. Statement = never use.

pgbouncer=true

Tells Prisma to pin connection for transaction duration. Required for $transaction in transaction mode.

RDS Proxy

AWS managed pooler. IAM auth, automatic failover, multiplexing. Best choice for ECS + RDS.

idle in transaction

Missing ROLLBACK in error paths. Use $transaction() to prevent — it handles ROLLBACK automatically.
