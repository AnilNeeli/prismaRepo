01

What is N+1?

You make 1 query to fetch a list, then N more queries — one for each item — to fetch related data. Total = N+1 database round trips instead of 1.

N+1 — bad

10 posts = 11 queries. 100 posts = 101 queries. Gets worse as data grows.

include — fixed

Always 2 queries regardless of how many posts. Prisma batches with IN clause.

1
SELECT * FROM "Post" — fetches all posts
2
SELECT * FROM "UserBlog" WHERE id = 'uuid-1' — loop query
3
SELECT * FROM "UserBlog" WHERE id = 'uuid-2' — loop query
4
SELECT * FROM "UserBlog" WHERE id = 'uuid-3' — loop query
02

Enable query logging — see it in terminal

First thing to do when debugging performance. Every SQL query Prisma sends will print in your terminal.

typescriptalways do this first
const prisma = new PrismaClient({
  log: ['query']  // prints every SQL query to terminal
})
What you'll see

Each query prints as prisma:query SELECT .... Count the lines — if you see N+1 lines for a loop, that's your problem. If you see 2 lines for an include, that's your fix working.

03

Reproduce N+1 deliberately

typescriptN+1 — the bad version
// 1 query — fetches all posts
const posts = await prisma.post.findMany()

// N queries — one per post — THIS IS THE PROBLEM
for (const post of posts) {
  const user = await prisma.userBlog.findUnique({
    where: { id: post.userId }
  })
  console.log(`${post.title} by ${user?.name}`)
}

// Terminal shows:
// prisma:query SELECT * FROM "Post"
// prisma:query SELECT * FROM "UserBlog" WHERE id = $1  ← repeated N times
// prisma:query SELECT * FROM "UserBlog" WHERE id = $1
// prisma:query SELECT * FROM "UserBlog" WHERE id = $1
04

Fix with include

Prisma batches all user IDs into a single IN clause — always 2 queries regardless of post count.

typescriptfixed — always 2 queries
const postsWithUsers = await prisma.post.findMany({
  include: { user: true }
})

for (const post of postsWithUsers) {
  console.log(`${post.title} by ${post.user.name}`)
}

// Terminal shows ONLY 2 queries:
// prisma:query SELECT * FROM "Post"
// prisma:query SELECT * FROM "UserBlog" WHERE id IN ('uuid-1', 'uuid-2', 'uuid-3')
Why include fixes it

Instead of one query per post, Prisma collects all the userId values and fires a single WHERE id IN (...) query. The query count stays fixed at 2 regardless of how many posts you have.

05

include vs select — full comparison

typescriptfour patterns
// Pattern 1 — all Post fields, no relation
prisma.post.findMany()

// Pattern 2 — all Post fields + full User record
prisma.post.findMany({
  include: { user: true }
})

// Pattern 3 — all Post fields + partial User (most common in production)
prisma.post.findMany({
  include: {
    user: { select: { name: true, email: true } }
  }
})

// Pattern 4 — partial Post + partial User (most performant)
prisma.post.findMany({
  select: {
    title: true,
    user: { select: { name: true, email: true } }
  }
})
Cannot use select + include together at top level

Prisma throws an error if you use both select and include at the same level. Use one or the other. You CAN use select nested inside include — that's Pattern 3 above.

PatternPost fieldsUser fieldsUse when
findMany() All None Don't need relations
include: { user: true } All All Dev / debugging only
include + nested select All Specific Production — need all post fields
select at top level Specific Specific Production — maximum performance
How select works on relations

When you list a relation field inside select, Prisma automatically knows to fetch it — no include needed. TypeScript also narrows the return type to exactly what you selected — accessing an unselected field is a compile-time error.

06

Nested N+1 and nested include

N+1 can happen at multiple levels. Posts → Users + Comments → Comment authors.

typescriptnested include — what it fetches
const posts = await prisma.post.findMany({
  include: {
    user: { select: { name: true } },     // who wrote the post
    comments: {
      include: {
        user: { select: { name: true } }  // who wrote each comment
      }
    }
  }
})

// Result shape:
// {
//   id, title, content, ...,
//   user: { name: 'Anil' },           ← post author
//   comments: [
//     { body: 'Great!', user: { name: 'User 2' } },   ← comment + author
//     { body: 'Thanks!', user: { name: 'User 3' } }
//   ]
// }

This runs 4 queries total — fixed, regardless of data size:

1
SELECT * FROM "Post"
2
SELECT name FROM "UserBlog" WHERE id IN (...) — post authors
3
SELECT * FROM "Comment" WHERE postId IN (...) — all comments
4
SELECT name FROM "UserBlog" WHERE id IN (...) — comment authors
07

include vs SQL JOIN

Both solve N+1 but work differently under the hood:

SQL JOIN — 1 query

One round trip. Data merged in Postgres. Fastest at the DB level. Use via $queryRaw when Prisma's include is too slow.

Prisma include — 2 queries

Two round trips. Prisma merges in Node.js memory. Simpler to write. Fine for 99% of cases.

ApproachQueriesWhen to use
Loop (N+1) 1 + N Never in production
include 2 (fixed) Most cases — simple and correct
Raw SQL JOIN 1 Complex reporting, maximum performance
08

The interview answer — memorize this flow

When asked "explain N+1 and how you fix it in Prisma"

1

Spot it

Enable log: ['query'], count the queries in terminal — if it grows with data, it's N+1

2

Cause

Fetching related data inside a loop — one query per iteration instead of batching

3

Fix

Use include — Prisma batches all IDs into a single IN clause, always 2 queries

4

Optimize

Use select inside include to avoid over-fetching large records

5

Edge case

Very large datasets where even the IN clause is slow → drop to $queryRaw with a real SQL JOIN

09

Full working file

typescriptsrc/day6.ts — complete
import 'dotenv/config'
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient({ log: ['query'] })

async function main() {

  // Seed data
  await prisma.userBlog.createMany({
    data: [
      { email: 'user1@test.com', name: 'User 1', age: 25 },
      { email: 'user2@test.com', name: 'User 2', age: 30 },
      { email: 'user3@test.com', name: 'User 3', age: 28 },
    ],
    skipDuplicates: true
  })

  // ❌ N+1 — watch query count in terminal
  console.log('\n--- N+1 (BAD) ---')
  const posts = await prisma.post.findMany()
  for (const post of posts) {
    const user = await prisma.userBlog.findUnique({ where: { id: post.userId } })
    console.log(`${post.title} by ${user?.name}`)
  }

  // ✅ Fixed with include + select
  console.log('\n--- FIXED WITH INCLUDE + SELECT ---')
  const postsFixed = await prisma.post.findMany({
    include: { user: { select: { name: true, email: true } } }
  })
  for (const post of postsFixed) {
    console.log(`${post.title} by ${post.user.name}`)
  }

  // ✅ Most performant — select at top level
  console.log('\n--- MOST PERFORMANT ---')
  const postsBest = await prisma.post.findMany({
    select: {
      title: true,
      user: { select: { name: true, email: true } }
    }
  })
  for (const post of postsBest) {
    console.log(`${post.title} by ${post.user.name}`)
  }

}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect())

N+1

1 query + N queries in a loop. Grows with data. Always bad in production.

Spot it

log: ['query'] in PrismaClient. Count lines in terminal. Fixed count = good.

Fix it

include batches into IN clause. Always 2 queries regardless of data size.

select vs include

Can't use both at top level. select inside include = partial fields on relation.

Nested include

Posts + authors + comments + comment authors = 4 queries. Fixed count.

include vs JOIN

include = 2 queries merged in Node. JOIN = 1 query in Postgres. Both fix N+1.
