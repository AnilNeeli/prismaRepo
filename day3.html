01

CRUD operations — overview

Create

  • create() — single record
  • createMany() — bulk insert

Read

  • findUnique() — by @id or @unique
  • findFirst() — first match, any field
  • findMany() — all matches

Update

  • update() — single by unique
  • updateMany() — multiple, returns count
  • upsert() — create or update

Delete

  • delete() — single by unique
  • deleteMany() — multiple, returns count
The key rule

findUnique / update / delete — where needs @id or @unique field. findMany / updateMany / deleteMany — where can use any field.

02

Create operations

typescriptcreate + createMany
// Single create — returns the created record
const user = await prisma.user.create({
  data: { email: 'anil@test.com', name: 'Anil' }
})
// user.id is auto-generated uuid — accessible immediately

// Nested create — User + Post in one query
await prisma.user.create({
  data: {
    email: 'test2@test.com',
    name: 'Test',
    posts: {
      create: { title: 'My first post', content: 'Hello' }
    }
  }
})

// Fetch with include after creation
const userWithPost = await prisma.user.findUnique({
  where: { email: 'test2@test.com' },
  include: { posts: true }
})

// createMany — bulk insert, no nested relations allowed
await prisma.tag.createMany({
  data: [
    { name: 'typescript' },
    { name: 'prisma' },
    { name: 'nodejs' },
  ],
  skipDuplicates: true  // don't throw if already exists
})
createMany limitation

createMany does not support nested relations — you can't create related records in the same call. It's for bulk inserting flat data only. Also doesn't return the created records, just a count.

03

Read operations

typescriptfindUnique · findFirst · findMany
// findUnique — only @id or @unique fields in where
// Returns: Post | null
const user = await prisma.user.findUnique({
  where: { email: 'anil@test.com' }  // email is @unique ✓
})

// findFirst — first match on any field, with ordering
// Returns: Post | null
const latestPost = await prisma.post.findFirst({
  where: { published: true },
  orderBy: { createdAt: 'desc' }
})

// findMany — all matching records
// Returns: Post[]
const posts = await prisma.post.findMany({
  where: { published: true },
  orderBy: { createdAt: 'desc' },
  take: 10,   // limit results
  skip: 0,    // offset — for pagination
})
MethodWhere clauseReturnsUse when
findUnique @id or @unique only T | null Fetching one record by ID or email
findFirst Any field T | null First match with ordering — latest post, oldest invoice
findMany Any field T[] All matches — lists, feeds, reports
Interview question

"What's the difference between findUnique and findFirst?" — findUnique only works on @id or @unique fields and Postgres can use an index lookup. findFirst works on any field but may do a full table scan. Use findUnique whenever you have a unique identifier.

04

Update operations

typescriptupdate · updateMany · upsert
// update — requires @id or @unique in where
// Throws P2025 if record not found
const updated = await prisma.post.update({
  where: { id: 'post-123' },
  data: { published: true }
})

// updateMany — any field in where, returns count
// Does NOT throw if nothing found — returns { count: 0 }
const { count } = await prisma.post.updateMany({
  where: { userId: user.id },
  data: { published: false }
})
console.log(`Updated ${count} posts`)

// upsert — create if not found, update if found
const tag = await prisma.tag.upsert({
  where: { name: 'typescript' },   // check if exists
  create: { name: 'typescript' },   // create if not found
  update: {}                           // nothing to update if found
})
update vs updateMany — critical difference

update() throws P2025 if the record doesn't exist. updateMany() silently returns count: 0. In fintech, prefer update() — you want to know if the record wasn't found rather than silently doing nothing.

Interview angle — upsert race condition

upsert is not atomic under high concurrency — two requests can both find "not exists" and both try to create, causing a unique violation. In fintech, use an explicit transaction with find-then-create instead of upsert for critical records.

06

Prisma input types — the typed client

When you run prisma generate, Prisma creates TypeScript types for every model. These are what make Prisma's type safety possible — mistakes are caught in your editor before they ever reach the DB.

PostCreateInput

Data for create(). Required fields are required, optional fields are optional. Uses relation objects.

PostUncheckedCreateInput

Same as CreateInput but uses raw FK values directly (userId: 'abc') instead of relation objects. Most common in real apps.

PostUpdateInput

Data for update(). Everything is optional — only pass what you want to change.

PostWhereInput

Filter for findMany(). Any field, supports AND/OR/NOT composition.

PostWhereUniqueInput

Filter for findUnique() / update() / delete(). Only @id and @unique fields allowed.

PostSelect / PostInclude

Control which fields and relations are returned. Use select to avoid over-fetching.

typescriptusing types in a repository
import { PrismaClient, Prisma } from '@prisma/client'

const prisma = new PrismaClient()

// PostCreateInput — required fields enforced by TypeScript
const createPost = async (data: Prisma.PostCreateInput) => {
  return prisma.post.create({ data })
}

// PostWhereUniqueInput — only @id or @unique
const getPost = async (where: Prisma.PostWhereUniqueInput) => {
  return prisma.post.findUnique({ where })
}

// PostWhereInput — any field combination
const getPosts = async (where: Prisma.PostWhereInput) => {
  return prisma.post.findMany({ where, orderBy: { createdAt: 'desc' } })
}

// PostUpdateInput — all optional, only change what you pass
const updatePost = async (
  where: Prisma.PostWhereUniqueInput,
  data: Prisma.PostUpdateInput
) => {
  return prisma.post.update({ where, data })
}

// TypeScript catches mistakes at compile time
createPost({ title: 123 })   // ❌ type error — title must be string
getPost({ userId: 'abc' })   // ❌ type error — userId is not @unique
CreateInput vs UncheckedCreateInput

PostCreateInput expects relation objects: user: { connect: { id: 'abc' } }. PostUncheckedCreateInput accepts raw FKs: userId: 'abc'. In real apps where you already have the FK (from a JWT token or URL param), use Unchecked — it's simpler and more common.

Three layers of validation

Prisma types are not the same as Joi validation — they catch errors at different stages. In a production fintech app you use all three:

Joi / Zod

Catches bad user input from outside

→
Prisma types

Catches developer mistakes at compile time

→
DB constraints

Catches race conditions and integrity violations

typescriptall three layers in a real endpoint
// Layer 1 — Joi validates incoming request body
const { error } = createPostSchema.validate(req.body)
if (error) return res.status(400).json({ error })

// Layer 2 — Prisma types ensure correct shape in your code
const post = await createPost(req.body as Prisma.PostCreateInput)

// Layer 3 — DB @unique constraint catches race conditions
// Prisma throws P2002 if two requests sneak through simultaneously
08

Error handling — P2002 and P2025

Prisma throws typed errors you can catch and handle gracefully. Two codes you must know cold:

P2002

Unique constraint violation

Thrown when you try to create a record that violates a @unique constraint — e.g. creating a user with an email that already exists.

P2025

Record not found

Thrown by update() and delete() when the record in the where clause doesn't exist.

typescripterror handling pattern
import { PrismaClient, Prisma } from '@prisma/client'

async function createUser(email: string) {
  try {
    return await prisma.user.create({ data: { email } })
  } catch (error) {
    if (error instanceof Prisma.PrismaClientKnownRequestError) {
      if (error.code === 'P2002') {
        throw new Error(`Email ${email} already taken`)
      }
      if (error.code === 'P2025') {
        throw new Error('Record not found')
      }
    }
    throw error  // rethrow unknown errors
  }
}
09

Mistakes made today

  • 1

    async const — invalid syntax

    The async keyword goes before the function body, not before const. Correct: const fn = async () => {}

  • 2

    Using userId in findUnique where clause

    findUnique only accepts @id or @unique fields. userId is a plain FK — not unique. Use findMany to fetch by userId.

  • 3

    prisma.userBlog doesn't exist

    Prisma generates client methods from the exact model name. Model is User → client method is prisma.user. Always match the model name exactly (camelCase).

  • 4

    Queries outside main() at top level

    Prisma queries are async and must run inside an async function. Top-level await requires ESM which conflicts with ts-node CommonJS setup. Keep all queries inside main().

  • 5

    Missing where and update in upsert

    upsert requires all three clauses: where (check if exists), create (what to create), update (what to update if found). Missing any one throws a type error.

Full working file

typescriptsrc/day3.ts — complete
import 'dotenv/config'
import { PrismaClient, Prisma } from '@prisma/client'

const prisma = new PrismaClient()

const createPost = async (data: Prisma.PostCreateInput) => {
  return prisma.post.create({ data })
}

async function main() {
  // Create
  const user = await prisma.user.create({
    data: { email: 'anil@test.com', name: 'Anil' }
  })

  // Nested create
  await prisma.user.create({
    data: {
      email: 'test2@test.com', name: 'Test',
      posts: { create: { title: 'My first post', content: 'Hello' } }
    }
  })

  // Fetch with include
  const userWithPost = await prisma.user.findUnique({
    where: { email: 'test2@test.com' },
    include: { posts: true }
  })

  // Bulk insert
  await prisma.tag.createMany({
    data: [{ name: 'typescript' }, { name: 'prisma' }, { name: 'nodejs' }],
    skipDuplicates: true
  })

  // Find
  const latestPost = await prisma.post.findFirst({
    where: { published: true }, orderBy: { createdAt: 'desc' }
  })

  // findMany by userId (non-unique field — use findMany not findUnique)
  const userPosts = await prisma.post.findMany({
    where: { userId: user.id }, orderBy: { createdAt: 'desc' }
  })

  // updateMany — returns count
  const { count } = await prisma.post.updateMany({
    where: { userId: user.id }, data: { published: false }
  })

  // upsert — all three clauses required
  const tag = await prisma.tag.upsert({
    where: { name: 'typescript' },
    create: { name: 'typescript' },
    update: {}
  })

  // deleteMany by userId
  await prisma.post.deleteMany({ where: { userId: user.id } })
}

main()
  .catch(console.error)
  .finally(() => prisma.$disconnect())

Day 3 summary

Unique rule

findUnique / update / delete need @id or @unique. Many variants accept any field.

upsert

Needs where + create + update. Not atomic under concurrency — use transaction in fintech.

Prisma types

CreateInput, UpdateInput, WhereInput, WhereUniqueInput. Import from Prisma namespace.

UncheckedCreateInput

Use raw FK values instead of relation objects. Most practical in real apps.

P2002

Unique constraint violation — email already exists, duplicate SKU, etc.

P2025

Record not found — thrown by update() and delete() when where has no match.
