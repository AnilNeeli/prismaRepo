01

The three raw SQL methods

$queryRaw

Read data, get typed results. Use TypeScript generic for type safety.

$executeRaw

Mutations — INSERT, UPDATE, DELETE. Returns number of affected rows.

Prisma.sql

Tagged template for safe interpolation. Parameterizes values automatically.

$queryRawUnsafe

Dynamic table/column names only. Never use with user input — SQL injection risk.

typescript$queryRaw — with TypeScript generic
// Without generic — returns unknown[]
const users = await prisma.$queryRaw`
  SELECT id, email, name FROM "UserBlog" WHERE age > 25
`

// With generic — fully typed
const users = await prisma.$queryRaw<
  { id: string; email: string; name: string }[]
>`
  SELECT id, email, name FROM "UserBlog" WHERE age > 25
`
// users[0].email is string — TypeScript knows
typescript$executeRaw — mutations
// Returns number of affected rows
const affected = await prisma.$executeRaw`
  UPDATE "UserBlog" SET "isActive" = false
  WHERE "lastLoginAt" < NOW() - INTERVAL '90 days'
`
console.log('Rows affected:', affected)
typescriptPrisma.sql — dynamic conditions
const minAge = 18
const status = 'ACTIVE'

// Prisma parameterizes ${} values automatically — SQL injection safe
const users = await prisma.$queryRaw`
  SELECT * FROM "UserBlog"
  WHERE age > ${minAge}
  AND status = ${status}
`

// Compose dynamic conditions
const condition = Prisma.sql`age > ${minAge}`
const users = await prisma.$queryRaw`
  SELECT * FROM "UserBlog" WHERE ${condition}
`
02

SQL injection safety

The tagged template syntax is safe because Prisma sends values as parameters, never inline SQL.

typescriptsafe — parameterized
const email = "admin@test.com' OR '1'='1"  // attempted injection

// ✅ Safe — Prisma sends as $1 parameter
const user = await prisma.$queryRaw`
  SELECT * FROM "UserBlog" WHERE email = ${email}
`
// Postgres sees: WHERE email = $1
// $1 = "admin@test.com' OR '1'='1" — treated as plain string, not SQL
typescriptnever do this — injection risk
// ❌ NEVER interpolate directly into $queryRawUnsafe
const email = req.body.email  // user input
await prisma.$queryRawUnsafe(
  `SELECT * FROM "UserBlog" WHERE email = '${email}'`
)
// Attacker sends: ' OR '1'='1 → returns all users

// ✅ $queryRawUnsafe only for hardcoded values you control
const tableName = 'UserBlog'  // hardcoded — not user input
await prisma.$queryRawUnsafe(`SELECT * FROM "${tableName}"`)
$queryRawUnsafe rule

Only use $queryRawUnsafe when you need dynamic table or column names — and only with hardcoded values you control, never user input. For dynamic values (user input, request params) always use the tagged template syntax.

03

When NOT to use Prisma ORM

The most common senior-level interview question. Have a crisp answer ready.

1

Complex aggregations — CTEs and window functions

Prisma groupBy is verbose and limited. CTEs and window functions are impossible. Drop to $queryRaw.

2

Bulk operations at scale

createMany has no nested relations and is slow for millions of rows. Raw SQL INSERT is orders of magnitude faster.

3

Full-text search

Postgres has native to_tsvector / plainto_tsquery support. Prisma only supports basic contains which uses ILIKE — not full-text.

4

Generated SQL is too slow

EXPLAIN ANALYZE shows sequential scan. Prisma's generated query can't be optimized further — drop to raw SQL for full control.

5

Query is simpler in SQL than Prisma

When the Prisma query API becomes more complex than the equivalent SQL — SQL wins on readability.

04

Real examples — Prisma vs raw SQL

typescriptwindow function — running total
// Impossible in Prisma ORM — must use $queryRaw
const result = await prisma.$queryRaw<
  { id: string; amount: number; runningTotal: number }[]
>`
  SELECT
    id,
    amount,
    SUM(amount) OVER (
      PARTITION BY "userId"
      ORDER BY "createdAt"
    ) as "runningTotal"
  FROM "Transaction"
  WHERE "userId" = ${userId}
`
typescriptCTE — multi-step query
// Prisma can't express CTEs — use $queryRaw
const result = await prisma.$queryRaw`
  WITH active_users AS (
    SELECT id FROM "UserBlog" WHERE "isActive" = true
  ),
  user_totals AS (
    SELECT "userId", SUM(amount) as total
    FROM "Transaction"
    WHERE "userId" IN (SELECT id FROM active_users)
    GROUP BY "userId"
  )
  SELECT u.email, ut.total
  FROM "UserBlog" u
  JOIN user_totals ut ON u.id = ut."userId"
  ORDER BY ut.total DESC
`
typescriptfull-text search
const results = await prisma.$queryRaw<
  { id: string; title: string; rank: number }[]
>`
  SELECT id, title,
    ts_rank(
      to_tsvector('english', title || ' ' || content),
      plainto_tsquery('english', ${searchTerm})
    ) as rank
  FROM "Post"
  WHERE to_tsvector('english', title || ' ' || content)
    @@ plainto_tsquery('english', ${searchTerm})
  ORDER BY rank DESC
  LIMIT 10
`
05

EXPLAIN ANALYZE — reading query plans

When a query is slow, run EXPLAIN ANALYZE to see what Postgres is actually doing under the hood.

typescripthow to run it
const plan = await prisma.$queryRaw`
  EXPLAIN ANALYZE
  SELECT * FROM "Post" WHERE "userId" = ${userId}
`
console.log(plan)

What to look for in the output:

BAD Seq Scan on "Post" — full table scan, no index used. Add @@index on that column.
GOOD Index Scan on "Post" — using an index. Query is efficient.
OK Bitmap Heap Scan on "Post" — partial index usage. Acceptable for low selectivity queries.
Workflow

Enable log: ['query'] → spot slow queries → run EXPLAIN ANALYZE → if Seq Scan on large table → add @@index in schema → if still slow → rewrite as $queryRaw with optimized SQL.

08

Day 12 summary

$queryRaw

Always add TypeScript generic for type safety. Default returns unknown[].

$executeRaw

Mutations only. Returns affected row count. No type generic needed.

SQL injection

Tagged template = safe. $queryRawUnsafe with user input = never. Prisma parameterizes ${} automatically.

Window functions

Prisma has zero native support. Always use $queryRaw. Common in fintech reporting.

EXPLAIN ANALYZE

Seq Scan = bad, needs index. Index Scan = good. Run when queries are slow.

When to escape

CTEs, window functions, bulk inserts, full-text search, slow generated SQL.
