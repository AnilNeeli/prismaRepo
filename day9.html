01

migrate dev vs migrate deploy

migrate dev — local only

  • Detects schema changes
  • Generates new SQL migration file
  • Applies migration to local DB
  • Regenerates Prisma Client
  • Runs seed script if configured
  • Resets DB if drift detected

migrate deploy — production

  • Checks _prisma_migrations table
  • Applies pending migrations in order
  • Never generates new files
  • Never runs seed script
  • Never resets DB
  • Safe for production
bashcommands
# Local development
npx prisma migrate dev --name add-age-to-user

# Production / CI-CD pipeline
npx prisma migrate deploy

# Check migration status
npx prisma migrate status
Interview question

"How do you handle migrations in your CI/CD pipeline?" — migrate deploy runs automatically on deploy, never migrate dev. migrate dev is local only — it can reset your DB if it detects drift, which would be catastrophic in production.

      migration.sql
Never edit a migration after applying it

Every migration file has a checksum stored in _prisma_migrations. If you edit the file after applying it, the checksum won't match and Prisma throws an error on next migrate deploy — blocking all future migrations. Serious production outage risk.

03

Shadow database

When you run migrate dev, Prisma needs to calculate what SQL to generate. It does this using a shadow database — a temporary clean database it creates and destroys automatically.

bashhow it works
Your schema.prisma (new state)
        ↓
Shadow DB (clean slate — Prisma creates + drops automatically)
Prisma replays ALL existing migrations on it → gets "before" state
        ↓
Diff: shadow DB state vs your current schema.prisma
        ↓
New migration SQL file generated from the diff

You never interact with the shadow database directly. It's why migrate dev needs two DB connections — one for your actual DB, one for the shadow.

Interview answer — shadow DB

Shadow DB is a temporary clean database Prisma creates and destroys automatically. It replays all existing migrations to get the "before" state, then diffs that against your current schema to generate the new migration SQL. You never touch it directly.

04

Zero-downtime migrations — expand-contract

The most important migration pattern for fintech interviews. Renaming a column directly locks the table and breaks running code.

sqldangerous — never do this in production
-- Locks table + breaks any code still using "fullName"
ALTER TABLE "User" RENAME COLUMN "fullName" TO "name";

The safe approach — two migrations + two code deployments:

Phase 1 — Expand

Add new column alongside old

Migration: add name column, backfill from fullName. Deploy code that writes to BOTH columns simultaneously.

Phase 2 — Contract

Remove old column

Deploy code that only uses name. Then migration: drop fullName. Never drop before updating code — old instances will break immediately.

prismatwo migrations in Prisma
// Migration 1 schema — add new alongside old
model User {
  fullName String?   // keep old column
  name     String?   // add new column
}

// Migration 2 schema — remove old column
model User {
  name String   // only new column remains
}
Interview answer — rename column with zero downtime

Use expand-contract: Migration 1 adds the new column and backfills data, deploy code that writes to both. Migration 2 drops the old column after code no longer references it. Never rename directly — it locks the table and breaks running instances.

05

Fixing failed migrations

A migration fails halfway through in production — Prisma marks it failed in _prisma_migrations and blocks all future migrations until resolved.

bashwhat you see
Error: P3009
migrate found failed migrations in the target database.
Migration 20250410_add_name_column failed.

# All future migrations blocked until resolved
1

Check status

npx prisma migrate status

2

Connect to DB and assess partial state

What was applied? What wasn't? Is data in a safe state?

3a

If reverting — undo SQL manually + resolve

npx prisma migrate resolve --rolled-back "migration_name"

3b

If completing — finish SQL manually + resolve

npx prisma migrate resolve --applied "migration_name"

4

Verify clean state

npx prisma migrate status → all migrations should show applied

Can you just re-run the migration?

Not directly — Prisma blocks re-runs on failed migrations. Most Prisma-generated SQL is not idempotent (ADD COLUMN without IF NOT EXISTS throws on second run). Fix: resolve --rolled-back, edit SQL to be idempotent, create new migration, deploy.

06

Adding indexes without locking — CONCURRENTLY

Normal CREATE INDEX takes an AccessExclusiveLock — no reads or writes until done. On a 10M row table this can take minutes, blocking your entire API.

sqldangerous on large tables
CREATE INDEX "User_email_idx" ON "User"("email");
-- Table locked — all queries blocked until index builds
sqlsafe — no lock
CREATE INDEX CONCURRENTLY "User_email_idx" ON "User"("email");
-- Table stays fully operational while index builds in background
bashhow to do it in Prisma
# Step 1 — generate SQL without applying
npx prisma migrate dev --name add-email-index --create-only

# Step 2 — edit the generated SQL file
# Change: CREATE INDEX → CREATE INDEX CONCURRENTLY

# Step 3 — apply
npx prisma migrate dev
ApproachTable lockBuild speedIn transactionUse when
Normal Yes — blocks all queries Faster Yes Small tables, new empty columns
CONCURRENTLY No lock 2-3x slower Not allowed Large tables in production
Failed CONCURRENTLY leaves INVALID index

If CREATE INDEX CONCURRENTLY fails midway, Postgres leaves a broken index marked as INVALID. You must DROP INDEX CONCURRENTLY the invalid one before retrying. Check with: SELECT indexname FROM pg_indexes WHERE indisvalid = false.

07

Idempotent SQL — safe to run multiple times

Idempotent means: run it once or run it ten times — the end result is always the same. Each line has a guard that says "only act if the work hasn't been done yet."

Not idempotent

  • ADD COLUMN "name"
  • UPDATE SET "name" = "fullName"
  • CREATE INDEX "idx"

Idempotent

  • ADD COLUMN IF NOT EXISTS "name"
  • UPDATE SET "name" = "fullName" WHERE "name" IS NULL
  • CREATE INDEX IF NOT EXISTS "idx"
sqldefensive migration — production pattern
-- Line 1: add column only if missing
ALTER TABLE "User" ADD COLUMN IF NOT EXISTS "name" TEXT;

-- Line 2: backfill only rows not yet migrated
UPDATE "User" SET "name" = "fullName" WHERE "name" IS NULL;

-- Line 3: add index without locking, skip if exists
CREATE INDEX CONCURRENTLY IF NOT EXISTS "User_name_idx" ON "User"("name");

What each guard does:

GuardProtects against
IF NOT EXISTS Column already created in a partial run — second run skips silently
WHERE "name" IS NULL Already-backfilled rows — only fills rows not yet migrated, never overwrites
CONCURRENTLY IF NOT EXISTS Index already built in partial run + no table lock on large tables

Day 10 summary
