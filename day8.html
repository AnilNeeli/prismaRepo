01

Nested write operations

connect

Link to an existing record by id or unique field

create

Create parent + child together in one call

connectOrCreate

Find existing and connect, or create if not found

disconnect

Remove relation link without deleting the record

update nested

Update child while updating parent in one call

Production rule

1 level deep nested write → fine in production. 2+ levels deep → break into separate explicit queries. Deeply nested writes are hard to debug and maintain.

02

connect — link to existing record

typescripttwo ways to connect
// Nested connect syntax
await prisma.post.create({
  data: {
    title: 'My post',
    user: { connect: { id: 'user-123' } }
  }
})

// Raw FK — simpler, same result
await prisma.post.create({
  data: {
    title: 'My post',
    userId: 'user-123'   // direct FK — use this in real apps
  }
})
Raw FK is simpler

When you already have the id (from a JWT token, URL param, or previous query), just pass the raw FK directly. The nested connect syntax is useful when you only have a unique field like email, not the id.

03

create — parent + children together

typescriptnested create
// Create user + 2 posts in one call
await prisma.userBlog.create({
  data: {
    email: 'anil@test.com',
    name: 'Anil',
    age: 25,
    posts: {
      create: [
        { title: 'Post 1' },
        { title: 'Post 2' },
      ]
    }
  }
})
P2002 on re-runs

If you run the same nested create twice, the email unique constraint throws P2002. Use a dynamic email or wrap in try/catch during dev. In production, always validate uniqueness before creating.

04

connectOrCreate — find or create

Use when you want to connect to a record that may or may not exist yet. In production, break complex nested connectOrCreate into explicit steps — easier to read and debug.

typescriptexplicit steps — production pattern
// Step 1 — find or create the tag
const tag = await prisma.tag.upsert({
  where: { name: 'typescript' },
  create: { name: 'typescript' },
  update: {}
})

// Step 2 — create the PostTag if it doesn't exist
await prisma.postTag.upsert({
  where: {
    postId_tagId: { postId: 'post-123', tagId: tag.id }
  },
  create: { postId: 'post-123', tagId: tag.id, taggedBy: 'anil@test.com' },
  update: {}
})

06

count + aggregate

typescriptcount · aggregate
// count — total records
const total = await prisma.post.count()

// count with filter
const published = await prisma.post.count({
  where: { published: true }
})

// aggregate — multiple stats in one query
const stats = await prisma.userBlog.aggregate({
  _count: { id: true },   // total users
  _avg:   { age: true },   // average age
  _min:   { age: true },   // youngest
  _max:   { age: true },   // oldest
  _sum:   { age: true },   // sum of all ages
})
console.log('Avg age:', stats._avg.age)
console.log('Total users:', stats._count.id)

_avgAVG(field)Numeric fields only
_sumSUM(field)Numeric fields only
_minMIN(field)Numeric + DateTime
_maxMAX(field)Numeric + DateTime

groupBy — aggregate per group

Groups records and runs aggregations per group. The SQL equivalent of GROUP BY.

What groupBy does — step by step

userIdcount
uuid-anil3

typescriptgroupBy + having
// Count posts per user, most posts first
const postCounts = await prisma.post.groupBy({
  by: ['userId'],
  _count: { id: true },
  orderBy: { _count: { id: 'desc' } }
})

// having — filter groups AFTER aggregation
// users with 2+ posts only
const activeUsers = await prisma.post.groupBy({
  by: ['userId'],
  _count: { id: true },
  having: {
    id: { _count: { gte: 2 } }
  }
})

// where + having together
// where filters rows BEFORE grouping
// having filters groups AFTER aggregation
const result = await prisma.post.groupBy({
  by: ['userId'],
  where: { published: true },          // only count published posts
  _count: { id: true },
  having: { id: { _count: { gte: 2 } } } // only groups with 2+ published posts
})
where vs having — interview question

where filters individual rows before grouping. having filters groups after aggregation. Example: where: { published: true } only counts published posts. having: { id: { _count: { gte: 2 } } } only returns groups where the count is 2 or more.

08

groupBy orderBy — why the extra nesting

Normal orderBy sorts by a column. groupBy orderBy sorts by a computed aggregation — so it needs one extra level to specify which aggregation and which field:

typescriptnormal vs groupBy orderBy
// Normal orderBy — sort by a column
orderBy: { createdAt: 'desc' }
// → ORDER BY "createdAt" DESC

// groupBy orderBy — sort by an aggregated value
orderBy: { _count: { id: 'desc' } }
// → ORDER BY COUNT("id") DESC

// Other aggregation sorts
orderBy: { _sum: { age: 'desc' } }   // ORDER BY SUM("age") DESC
orderBy: { _avg: { age: 'asc' } }   // ORDER BY AVG("age") ASC
Pattern to remember

Normal orderBy: { field: direction }. groupBy orderBy: { aggregation: { field: direction } } — one extra level because you're sorting by a computed value, not a raw column.

09

When to use $queryRaw instead

For complex aggregations, raw SQL is often cleaner. This is a legitimate answer to "when would you NOT use Prisma?"

typescriptPrisma groupBy — verbose
await prisma.post.groupBy({
  by: ['userId'],
  _count: { id: true },
  having: { id: { _count: { gte: 2 } } },
  orderBy: { _count: { id: 'desc' } }
})
typescript$queryRaw — cleaner + typed
const result = await prisma.$queryRaw<
  { userId: string; post_count: number }[]
>`
  SELECT "userId", COUNT("id")::int as post_count
  FROM "Post"
  GROUP BY "userId"
  HAVING COUNT("id") >= 2
  ORDER BY post_count DESC
`
// result[0].userId, result[0].post_count — fully typed
Window functions

Prisma has no native support for window functions (OVER PARTITION BY). Always use $queryRaw for them. This is also a valid interview answer — "for window functions and complex reporting I drop to $queryRaw and write SQL directly."

10

Exercise — mistakes + corrections

  • 1

    _count: { userId: true } — wrong field

    Count by id not userId. You're counting records, not foreign keys. Use _count: { id: true }.

  • 2

    orderBy: { _count: 'desc' } — missing field

    groupBy orderBy needs the field name inside the aggregation: orderBy: { _count: { id: 'desc' } }.

  • 3

    Used aggregate instead of groupBy for Q4

    aggregate doesn't support by or having — those are groupBy options. Use prisma.post.groupBy() with having to filter groups.

Day 8 summary

connect

Link to existing record. Use raw FK directly when you have the id — simpler than nested syntax.

nested create

Parent + children in one call. Max 1 level deep in production. Use explicit steps beyond that.

aggregate

_count, _avg, _sum, _min, _max. Multiple stats in one query. Numeric fields only for math ops.

groupBy

Aggregate per group. by: ['field'] groups records. _count per group. orderBy the aggregation.

where vs having

where filters rows before grouping. having filters groups after aggregation.

$queryRaw

Complex aggregations and window functions. SQL is cleaner. Add TypeScript generic for type safety.
